* Ownership

This repo owns the human-edited configuration. Generated state stays outside the
repo and should not be edited directly.

** System layer

File:

#+begin_src text
guix/systems/guixpc.scm
#+end_src

Owns:

- kernel and boot arguments
- bootloader
- filesystems
- users and groups
- system packages
- system services such as SSH, Bluetooth, Xorg, elogind

Apply with:

#+begin_src sh
gsr
#+end_src

** Home layer

File:

#+begin_src text
guix/home/guixpc.scm
#+end_src

Owns:

- user-level Guix Home services
- user-level service profile
- session services such as DBus, PipeWire, WirePlumber, PipeWire-Pulse
- live symlinks from selected dotfiles back into this repo
- Fish interactive configuration and native prompt

Apply with:

#+begin_src sh
ghr
#+end_src

Current live pointer:

#+begin_src text
~/.config/guix/home.scm -> ~/Projects/guix-dotfiles/guix/home/guixpc.scm
~/.guix-home             -> generated Guix Home profile in /gnu/store
#+end_src

Do not edit =~/.guix-home= directly.

** Application config

Files:

#+begin_src text
emacs/
kitty/
ranger/
gdb/
guile/
shell/
x11/
assets/
#+end_src

Owns normal dotfile behavior. Most changes here apply by reloading the affected
program. A Guix rebuild is only needed when a package or service declaration
changes.

Guix Home recreates the live app symlinks to these files during =ghr=. It does
not replace them with store snapshots, so normal repo edits remain live.

Fish behavior lives in =shell/config.fish= and is installed through Guix Home
because Fish is the account login shell. Run =ghr= after editing it. The prompt
is native Fish, not Starship. Fastfetch is used as an interactive shell greeter,
so it appears in kitty or any other terminal that starts an interactive Fish shell.

* Emacs and EXWM baseline

This machine uses Emacs as both editor and X window manager.

** Startup path

#+begin_src text
Display manager / X session
-> Guix emacs-exwm wrapper
-> ~/.exwm
-> emacs/exwm-loader.el
-> ~/.emacs.d/exwm.el
-> emacs/exwm.el

Regular Emacs init:
~/.emacs.d/init.el -> emacs/init.el
#+end_src

The live EXWM session should report EXWM as loaded:

#+begin_src sh
emacsclient --eval "(list :server t :daemon (daemonp) :user-init user-init-file :exwm-loaded (featurep (quote exwm)) :server-running (server-running-p))"
#+end_src

Batch-load the editor config without starting an EXWM session:

#+begin_src sh
cd ~/Projects/guix-dotfiles
emacs --batch --eval "(progn (load-file \"emacs/init.el\") (message \"sk init batch loaded\"))"
#+end_src

** EXWM app launcher

=s-SPC= opens a Counsel/Ivy app launcher from installed desktop entries.

The first implementation tried to delegate launching to =gtk-launch=, matching
what =counsel-linux-app= does upstream. On this Guix system neither
=gtk-launch= nor =gtk4-launch= is present in the active profile, so the local
launcher keeps those as optional fast paths and otherwise launches directly from
the selected =.desktop= file:

1. resolve the selected desktop id, for example =blueman-manager.desktop=
2. find the desktop file under the application directories
3. read the =Exec== field from the =[Desktop Entry]= section
4. remove desktop-entry field codes such as =%U=, =%F=, and =%i=
5. launch the command asynchronously with Emacs =start-process=

Terminal desktop entries are launched through =kitty -- ...=. This keeps GUI
app launching usable without adding another standalone launcher while preserving
the documented desktop-entry model.

Reference points:

- Desktop Entry Specification, =Exec= field codes:
  https://specifications.freedesktop.org/desktop-entry-spec/latest/exec-variables.html
- Emacs asynchronous process launch:
  https://www.gnu.org/software/emacs/manual/html_node/elisp/Asynchronous-Processes.html

Bluetooth Manager can be tested through the launcher by selecting
=Bluetooth Manager=, or directly with:

#+begin_src sh
blueman-manager
#+end_src

If Blueman opens with icon warnings, that is a GTK/icon-theme issue, not the
app launcher path.

** Keyboard repeat

Keyboard repeat is intentionally aligned with the MSI profile:

#+begin_src text
delay: 210 ms
rate:  67 repeats/sec requested
#+end_src

EXWM applies this with:

#+begin_src sh
xset r rate 210 67
#+end_src

=xset= may report the rate as =71= because Xorg rounds the interval internally;
the requested value remains =67=. Validate the live setting with:

#+begin_src sh
xset q | grep "auto repeat delay"
#+end_src

* Apply

Use the smallest command that matches the changed layer.

System changes:

#+begin_src sh
gsr
#+end_src

Home service changes:

#+begin_src sh
ghr
#+end_src

Both layers changed:

#+begin_src sh
gsr
ghr
#+end_src

Regular application config changes usually need only a program reload.

* Live symlinks

The important live files are symlinks back into this repo:

#+begin_src text
~/.config/guix/channels.scm       -> guix/channels.scm
~/.config/guix/home.scm           -> guix/home/guixpc.scm
~/.config/guix/systems/guixpc.scm -> guix/systems/guixpc.scm
~/.emacs.d/init.el                -> emacs/init.el
~/.emacs.d/exwm.el                -> emacs/exwm.el
#+end_src

The app and shell symlinks are managed by =guix/home/guixpc.scm=. The Guix
entrypoint symlinks under =~/.config/guix= are bootstrap links and intentionally
stay outside the Guix Home activation service so =gsr= and =ghr= always read the
editable repo files.

Guix Home recreates the live app symlinks to these files during =ghr=. It does
not replace them with store snapshots, so normal repo edits remain live.
=scripts/guix-reconfigure= now runs a read-only check of all 15 links through
the fixed System-profile Guile before it invokes Guix.  A regular-file,
directory, dangling-link, or missing-source conflict present at that preflight
is preserved and prevents the built-in Home activation services from starting.
The activation service repeats the same complete validation before changing any
project link.  A concurrent path race can still reach that later boundary after
earlier built-in activation, so =ghr= remains the supported path and the
additive System fallback is retained throughout this checkpoint.  Invoking
=guix home reconfigure= directly bypasses the early wrapper preflight.

Fish behavior lives in =shell/config.fish= and is installed through Guix Home
because Fish is the account login shell. Run =ghr= after editing it. The prompt
is native Fish, not Starship. Fastfetch is used as an interactive shell greeter,
so it appears in kitty or any other terminal that starts an interactive Fish
shell.  Direct SSH commands start Fish as non-login; when the Home profile is
not first in PATH, the Fish config imports =~/.profile= through the explicitly
owned =fish-foreign-env= package.  Local child Fish processes keep their
inherited environment unchanged.

There is no graphical display manager in the intended path.  The system config

The session runner keeps a bounded private current/previous log pair:

#+begin_src text
~/.local/state/sk-exwm/startx-current.log
~/.local/state/sk-exwm/startx-previous.log
#+end_src

If =XDG_STATE_HOME= is set, its value replaces =~/.local/state=.
The previous file appears after a legacy-log migration or the second attempt;
the state directory never grows beyond the pair.

A failed or normally ended X session returns to the existing tty1 Fish prompt
instead of terminating the login shell and triggering another mingetty
autologin.  Inspect the current log, correct the cause, and run =sk-start-exwm=
to retry.  SSH, an existing graphical display, and non-tty1 login shells remain
silent and never autostart X.

The checker has two independent phases:

The full Guix build gate also calls the batch phase against the newly built
Home and System profiles before activation:

#+begin_src sh
SK_EMACS_CHECK_HOME_PROFILE=/gnu/store/...-home/profile \
SK_EMACS_CHECK_SYSTEM_PROFILE=/gnu/store/...-system/profile \
  scripts/emacs-check --batch-only

Candidate mode deliberately skips the active EXWM server, runs the exact Home
Emacs executable, and exposes only the candidate Home/System =PATH= and
=EMACSLOADPATH=.  It is an internal pre-activation proof; the ordinary command
above remains the combined live-and-isolated acceptance check.

The live phase never loads configuration.  It checks the running executable,
EXWM/server state, package generations, hooks, keys, and positive load-path
ownership.  The batch phase prevents writes outside its temporary root, keeps
server/history/notes startup disabled, captures warnings, runs real mode hooks
with only deferred LSP process startup stubbed, and executes ERT, formatter,
byte-compile, and Checkdoc gates.  Repository status is compared before and
after both phases.

These maintenance-only negative controls must each exit nonzero while reporting
the other phase independently:

#+begin_src sh

SK_EMACS_CHECK_BREAK_BATCH=1 scripts/emacs-check
SK_EMACS_CHECK_BREAK_WARNING=1 scripts/emacs-check
#+end_src

P2.1 adds a separate observational performance snapshot.  It requires Linux
=/proc= and clocksource sysfs, Python 3.10 or newer, Git with an upstream, a clean
synchronized checkout, converged booted/current/profile System links, and the
exact active-Home =emacsclient= and EXWM server.  Close every other
=emacsclient= or =.emacsclient-real= process and remain idle during the normally
short, per-call-bounded timing window:

#+begin_src sh
cd ~/Projects/guix-dotfiles
scripts/emacs-performance-snapshot
#+end_src

The command emits versioned TSV to stdout.  It records live/fixture mode,
runtime and source provenance, current-process initialization/start/uptime, GC
and memory state, 21 monotonic host-local server-RPC samples, and five
preparation-only launcher samples with raw values and descriptive distribution
statistics.  The launcher probe replays discovery/filtering and stops before
Ivy, application execution, or window placement.  The sampler refuses dirty or
unsynchronized source, client/profile mismatch, another client process, or
protected state drift.  It does not force garbage collection, persistently
clear or warm the Counsel cache, tune Emacs, reload the config, activate a
profile, or reboot.  Its evaluations still allocate and may advance cumulative
GC/memory observations or warm the kernel filesystem cache.  See
[[file:docs/audits/2026-07-15-p2.1-emacs-exwm-measurement.org][the P2.1 protocol and truth boundaries]] before interpreting its numbers.

Publish the implementation checkpoint before collecting evidence.  From the
then-clean =0/0= checkout, redirect stdout to
=/tmp/guixpc-p2.1-emacs-exwm-v1.tsv= and stderr to the adjacent =.stderr= file.
Require exit zero and empty stderr; review =run_mode= =live= metadata, sample
counts, invariants, and SHA-256.  Only then add the unchanged TSV in a separate
evidence commit and record its digest/interpretation in the P2.1 Evidence
section.  Redirecting directly into the checkout correctly fails the clean-tree
preflight.

** Compositor

Picom is intentionally scoped to Emacs frame transparency only:

#+begin_src text
picom --config /dev/null --backend glx --vsync --opacity-rule '85:class_g = "Emacs"'
#+end_src

The X session starts the wallpaper and Picom before launching Emacs/EXWM so the

** Application placement and reload policy

Desktop entries are fully resolved and validated before a process starts.  The
launcher omits =Hidden=/=NoDisplay= entries and conservatively omits complex
=Exec= quoting that the direct parser cannot reproduce faithfully.  It never
creates a speculative split.  Instead, a 30-second launch intent records the
origin workspace, process PID, and normalized class metadata.  The EXWM manage
hook rejects transient, dialog, fixed-size, and floating clients; it prefers
exact/descendant PID matching and allows live class fallback only for the known
single-instance Chromium path.  An ambiguous match is ignored.  Failure,
timeout, successful placement, and reload all remove the intent and timer.

Configuration reload reader-checks every module before loading the first one.
It reports the exact failed module and how many earlier modules loaded, restores
the complete project display-policy state, and explicitly warns that arbitrary
earlier function/hook mutations cannot be rolled back.  Project display rules
are identity-owned and preserve foreign/package rules.  The reviewed runtime
floor is EXWM 0.35; its remaining private workspace/floating adapters are
guarded by that exact compatibility check.  The Xref RET adapter is reviewed on
Emacs 30 and degrades with a warning instead of blocking editor startup on a
future Emacs major release.

** EXWM app launcher

=s-SPC= opens a Counsel/Ivy app launcher from installed desktop entries.

The first implementation tried to delegate launching to =gtk-launch=, matching
what =counsel-linux-app= does upstream. On this Guix system neither
=gtk-launch= nor =gtk4-launch= is present in the active profile, so the local
launcher keeps those as optional fast paths and otherwise launches directly from
the selected =.desktop= file:

1. resolve the selected desktop id, for example =blueman-manager.desktop=
2. find the desktop file under the application directories
3. read the =Exec== field from the =[Desktop Entry]= section
4. remove desktop-entry field codes such as =%U=, =%F=, and =%i=
5. launch the command asynchronously with Emacs =start-process=

Terminal desktop entries are launched through =kitty -- ...=. This keeps GUI
app launching usable without adding another standalone launcher while preserving
the documented desktop-entry model.

Reference points:

- Desktop Entry Specification, =Exec= field codes:
  https://specifications.freedesktop.org/desktop-entry-spec/latest/exec-variables.html
- Emacs asynchronous process launch:
  https://www.gnu.org/software/emacs/manual/html_node/elisp/Asynchronous-Processes.html

Bluetooth Manager can be tested through the launcher by selecting
=Bluetooth Manager=, or directly with:

#+begin_src sh

#+end_src

The accepted Slice 19 native-comp policy keeps Guix Home as the sole Org package
owner and moves writable native artifacts out of =~/.emacs.d=.  The cache key
is the immutable store basename reached through =~/.guix-home/profile=, not the

can reuse only its matching cache.  =emacs/early-init.el= also removes the old
=~/.emacs.d/eln-cache/= entry from fallback lookup; merely prepending a new
directory would not isolate a cold start from stale native files.  Do not
suppress Org's version assertion or install a second Org through ELPA.

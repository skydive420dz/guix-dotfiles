* Ownership

This repo owns the human-edited configuration. Generated state stays outside the
repo and should not be edited directly.

** System layer

File:

#+begin_src text

#+end_src

- kernel and boot arguments
- bootloader
- filesystems
- users and groups
- system packages
- system services such as SSH, Bluetooth, Xorg, elogind

** Home layer

** Application config

Files:

#+begin_src text
emacs/
kitty/
ranger/
gdb/
guile/
shell/
x11/
assets/
#+end_src

Owns normal dotfile behavior. Most changes here apply by reloading the affected
program. A Guix rebuild is only needed when a package or service declaration
changes.

Guix Home recreates the live app symlinks to these files during =ghr=. It does
not replace them with store snapshots, so normal repo edits remain live.

Fish behavior lives in =shell/config.fish= and is installed through Guix Home
because Fish is the account login shell. Run =ghr= after editing it. The prompt
is native Fish, not Starship. Fastfetch is used as an interactive shell greeter,
so it appears in kitty or any other terminal that starts an interactive Fish shell.

* Emacs and EXWM baseline

This machine uses Emacs as both editor and X window manager.

** Startup path

#+begin_src text
tty1 mingetty autologin
-> Fish login shell
-> shell/config.fish guard
-> startx
-> ~/.xinitrc
-> shell/xinitrc
-> exwm
-> ~/.exwm
-> emacs/exwm-loader.el
-> ~/.emacs.d/exwm.el
-> emacs/exwm.el

Regular Emacs init:
~/.emacs.d/init.el -> emacs/init.el
#+end_src

There is no graphical display manager in the intended path.  The system config
removes GDM, tty1 autologins as =skydive420dz=, and Fish starts EXWM only for
the local interactive tty1 login shell.  SSH shells are excluded by the Fish
guard.

The live EXWM session should report EXWM as loaded:

#+begin_src sh
emacsclient --eval "(list :server t :daemon (daemonp) :user-init user-init-file :exwm-loaded (featurep (quote exwm)) :server-running (server-running-p))"
#+end_src

Validate the editor config with the same Guix profile assumptions used by the
live session:

#+begin_src sh
cd ~/Projects/guix-dotfiles
scripts/emacs-check
#+end_src

Do not use plain =emacs --batch= from SSH as the canonical check. Noninteractive
SSH may start with a minimal =PATH= and no =EMACSLOADPATH=, which can produce
false package warnings even when the live EXWM Emacs session can load the same
Guix-provided packages.

** Emacs language acceptance

Language support is split by backend. LSP-backed languages use =lsp-mode=,
Company, LSP UI, and =SPC c x= through the LSP diagnostics path. Non-LSP
languages use their native Emacs package plus Flycheck or REPL tooling where
appropriate. =SPC c x= dispatches by backend: LSP diagnostics for LSP buffers,
Flycheck errors for non-LSP buffers.

#+begin_src text
Lua
  Mode:        lua-mode
  Backend:     lsp-mode + lua-language-server
  Diagnostics: LSP/Flycheck path
  Notes:       LSP-backed language model baseline.

Python
  Mode:        python-mode
  Backend:     lsp-mode + python-lsp-server / pylsp
  Diagnostics: LSP/Flycheck path
  Notes:       Pyright is not required for the current baseline.

Shell
  Mode:        sh-mode
  Backend:     Flycheck sh-bash + shellcheck
  Diagnostics: Flycheck path
  Notes:       shfmt is installed for formatting, but no format-on-save policy.

JSON
  Mode:        json-mode when available; built-in js-json-mode fallback
  Backend:     Flycheck json-jq
  Diagnostics: Flycheck path
  Notes:       jq is preferred over Flycheck's python checker because this Guix
               profile exposes python3, while that checker looks for python.

Org
  Mode:        org-mode
  Backend:     Org native features
  Diagnostics: none
  Notes:       Owns notes, capture, agenda, source block helpers, and editing UI.

C
  Mode:        c-mode
  Backend:     lsp-mode + clangd from the clang package
  Diagnostics: LSP/Flycheck path
  Notes:       clang-format is installed, but no format-on-save policy.

Emacs Lisp
  Mode:        emacs-lisp-mode
  Backend:     Emacs native Eldoc/help/eval/navigation
  Diagnostics: none currently
  Notes:       Not LSP-backed.

Scheme / Guile
  Mode:        scheme-mode
  Backend:     Geiser + Geiser Guile
  Diagnostics: Geiser/native interaction
  Notes:       Guile is installed system-wide.

Common Lisp
  Mode:        lisp-mode for .lisp, .cl, and .asd
  Backend:     SLY + SBCL when SLY is started
  Diagnostics: SLY/native interaction
  Notes:       File opening is owned by lisp-mode; SLY is the REPL/tooling layer.
#+end_src

Acceptance probes should check actual buffer state, not just package presence:

#+begin_src text
major mode
backend mode or REPL integration
completion frontend when applicable
diagnostics path when applicable
formatter availability when intentionally installed
#+end_src

Do not mark a language complete because a package exists. The buffer has to open
in the intended mode and the selected diagnostics/completion path has to work.

** EXWM keys

#+begin_src text
s-SPC                 open app launcher from desktop entries
s-RET                 open kitty
s-w                   open Chromium
s-r                   reload EXWM config
s-h / s-j / s-k / s-l focus window left/down/up/right
s-H / s-J / s-K / s-L move current window left/down/up/right
s-q                   close current EXWM app window; delete Emacs window otherwise
s-b                   switch buffer
s-f                   toggle fullscreen for current EXWM app window
s-m                   promote selected window to master
s-M                   normalize visible windows into master/stack layout
s-1 .. s-5            switch/create EXWM workspace 1 through 5
s-! s-@ s-# s-$ s-%   move current EXWM app window to workspace 1 through 5
XF86AudioRaiseVolume  raise default sink volume through wpctl
XF86AudioLowerVolume  lower default sink volume through wpctl
#+end_src

=s-h= uses the normal left-window focus path first. If normal windmove cannot
see a left target and Treemacs is open, it falls back to selecting Treemacs.

Workspace numbers are user-facing and one-based. EXWM internally uses zero-based
workspace indexes, so =s-1= maps to EXWM workspace index 0, =s-2= maps to index
1, and so on.

** Emacs window policy

Window behavior is intentionally owned by the shared Emacs window policy, not by
each package independently.

#+begin_src text
emacs/lisp/sk-window-policy.el
  Owns master/stack layout, side panels, transient panels, full-frame toggles,
  stale side-window cleanup, and helper command wrappers.

emacs/lisp/sk-exwm.el
  Owns EXWM startup, external app launch, workspaces, and global =s-*= keys.
  It delegates layout decisions to =sk-window-policy.el=.

emacs/lisp/sk-keys.el
  Owns user-facing keybindings. Commands that affect layout should call the
  policy wrappers instead of raw package commands.
#+end_src

The intended layout contract:

#+begin_src text
Master/main area:
  normal files and selected results open here

Left persistent panel:
  Treemacs project tree

Right utility panel:
  Dired
  Ibuffer
  Magit status
  Xref/reference results
  Help/Helpful/Eldoc
  Eshell
  Vterm

Bottom transient panel:
  Which-Key
  Warnings / Compile logs
#+end_src

The =SPC o= menu should stay compact and predictable:

#+begin_src text
SPC o d  Dired, right utility panel
SPC o e  Eshell, right utility panel
SPC o t  Treemacs project tree, left persistent panel
SPC o v  Vterm, right utility panel
#+end_src

=term= is still available through =M-x term=, but the leader binding is reserved
for Treemacs. Vterm is the preferred terminal buffer once it is available on
this machine.

File-opening commands are intentionally split by interaction style:

#+begin_src text
SPC o d  Dired panel for visual browsing and file operations
SPC f f  Counsel find-file for path-based opening
SPC .    Counsel FZF for fast fuzzy opening
SPC p f  Counsel Projectile find-file for project fuzzy opening
#+end_src

Selector/helper commands should keep their helper panel open and open selected
content in the master/main area:

#+begin_src text
Dired RET:
  open selected file in master/main window
  keep Dired panel open

Ibuffer RET:
  open selected buffer in master/main window
  keep Ibuffer panel open

Xref RET:
  jump selected result in master/main window
  keep Xref results panel open

Treemacs RET / l:
  open selected file/location in master/main window
  keep Treemacs panel open
#+end_src

Do not override Magit internals without a specific reason. Magit has a large
surface of transient and action buffers; start by controlling where Magit opens,
then only adjust internal actions if the actual behavior is wrong.

With Evil collection active, Magit refresh is =g r=. Plain =g= is an Evil
prefix and will show the available =g= commands through Which-Key.

** EXWM app launcher

=s-SPC= opens a Counsel/Ivy app launcher from installed desktop entries.

The first implementation tried to delegate launching to =gtk-launch=, matching
what =counsel-linux-app= does upstream. On this Guix system neither
=gtk-launch= nor =gtk4-launch= is present in the active profile, so the local
launcher keeps those as optional fast paths and otherwise launches directly from
the selected =.desktop= file:

1. resolve the selected desktop id, for example =blueman-manager.desktop=
2. find the desktop file under the application directories
3. read the =Exec== field from the =[Desktop Entry]= section
4. remove desktop-entry field codes such as =%U=, =%F=, and =%i=
5. launch the command asynchronously with Emacs =start-process=

Terminal desktop entries are launched through =kitty -- ...=. This keeps GUI
app launching usable without adding another standalone launcher while preserving
the documented desktop-entry model.

Reference points:

- Desktop Entry Specification, =Exec= field codes:
  https://specifications.freedesktop.org/desktop-entry-spec/latest/exec-variables.html
- Emacs asynchronous process launch:
  https://www.gnu.org/software/emacs/manual/html_node/elisp/Asynchronous-Processes.html

Bluetooth Manager can be tested through the launcher by selecting
=Bluetooth Manager=, or directly with:

#+begin_src sh
blueman-manager
#+end_src

If Blueman opens with icon warnings, that is a GTK/icon-theme issue, not the
app launcher path.

** Keyboard repeat

Keyboard repeat is intentionally aligned with the MSI profile:

#+begin_src text
delay: 210 ms
rate:  67 repeats/sec requested
#+end_src

EXWM applies this with:

#+begin_src sh
xset r rate 210 67
#+end_src

=xset= may report the rate as =71= because Xorg rounds the interval internally;
the requested value remains =67=. Validate the live setting with:

#+begin_src sh
xset q | grep "auto repeat delay"
#+end_src

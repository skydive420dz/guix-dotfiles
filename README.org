#+title: GuixPC configuration
#+startup: overview

* Purpose

This repository is the human-edited source of truth for Rafael Oliveira's
=guixpc=.  It defines a Guix System, a Guix Home environment, and the EXWM
desktop used on the machine every day.

The goal is a coherent, reliable, responsive, and maintainable workstation
built around Guix, EXWM, Guile Scheme, and Emacs Lisp.  It is not a showcase
for collecting language dialects, a research platform, or permission to build
infrastructure without a practical benefit to the running system.

* Project contract

** Work from explicit outcomes

Every implementation slice must identify:

- the user objective it serves;
- the observable result Rafael should receive;
- the files and live surfaces it may affect;
- the smallest relevant validation;
- its stopping point.

Adjacent features, packages, languages, abstractions, and experiments are out
of scope until Rafael explicitly accepts them.  A =TODO=, =BACKLOG=, test, or
existing source file does not authorize work by itself.

If a slice expands materially, stops producing practical progress, or Rafael
questions its value, stop and reassess.  Report what changed, what has reached
the live machine, what remains, and the realistic cost before continuing.  Do
not answer a scope or value concern by persuading Rafael to endure more work.

** Use native Lisp for custom behavior

Custom behavior belongs in the native Lisp for its layer:

- Guile Scheme for Guix declarations, services, system utilities, and
  standalone tools;
- Emacs Lisp for Emacs and EXWM behavior.

Existing upstream software may use other languages.  Shell is acceptable only
as small boundary glue when the operating-system interface requires it; the
feature logic should remain in Scheme or Emacs Lisp.  Do not add another Lisp
dialect merely for breadth.  Any new dialect or non-Lisp implementation needs
a concrete requirement and Rafael's approval.

** Prove the layer that changed

Keep these claims separate:

1. source is valid;
2. a candidate builds;
3. a configuration was activated;
4. the expected runtime owns the new state;
5. Rafael completed any necessary hand test.

A build, test suite, document, or generated artifact is not a live system
improvement.  Call work complete only at the boundary Rafael requested.

** Accepted operating baseline

- Guix System owns hardware, boot, filesystems, system services, and a small
  tty recovery floor.
- Guix Home owns the normal user package set, Fish, Emacs, session D-Bus,
  PipeWire/WirePlumber, fonts, themes, and repository-backed dotfile links.
- EXWM is the desktop session; tty1 remains the local recovery entry point.
- Kitty and its terminfo are available in the System recovery floor.
- Interactive SSH carries the truthful =xterm-kitty= terminal type without a
  manual =TERM= override.
- GTK 3/4, Emacs, Kitty, Fish, wallpaper, fonts, icons, and cursor share the
  accepted dark visual policy.  Qt normalization has not yet been designed.
- Bluetooth is enabled early enough for a paired device to reconnect, while
  the user audio stack remains owned by Guix Home.
- Channel inputs are authenticated and pinned in =guix/channels.scm=.
- Generation retention inspection is read-only.  There is no approved
  automatic System-generation deletion or store garbage-collection job.

This is a concise baseline, not a substitute for live verification after a
change.

The repository owns human-edited configuration.  Guix store outputs,
generations, caches, application history, and other generated state are not
source files and should not be edited as if they were configuration.

** Live declaration links

The supported Guix entry points are links into this checkout:

#+begin_src text
~/.config/guix/channels.scm       -> guix/channels.scm
~/.config/guix/home.scm           -> guix/home/guixpc.scm
~/.config/guix/systems/guixpc.scm -> guix/systems/guixpc.scm
#+end_src

Inspect them without mutation:

#+begin_src sh
scripts/guix-entrypoints check
#+end_src

=scripts/guix-entrypoints bootstrap= may create or repair those three links,
but it must be used only for an explicit bootstrap task.  It refuses to
overwrite conflicting regular files or directories.

* Normal workflow

Run commands from =~/Projects/guix-dotfiles= on =guixpc=.

** Establish the boundary

Before editing:

#+begin_src sh
git status --short --branch
scripts/guix-entrypoints check
#+end_src

Confirm which owner is changing:

- application or Emacs configuration may need only a reload;
- Home packages, services, generated themes, Fish, or managed links require a
  Home reconfiguration;
- kernel, boot, filesystems, recovery packages, or system services require a
  System reconfiguration and usually a reboot.

Do not turn a local application change into a System or Home rebuild without a
real ownership reason.

** Preview reconfiguration

Fish exposes two aliases backed by =scripts/guix-reconfigure=:

#+begin_src sh
ghr --print                       # Guix Home plan
gsr --print                       # Guix System plan
#+end_src

The wrapper checks the declaration links, channel lock, source path, Guix
frontend, and relevant preflight before invoking Guix.  Direct =guix home
reconfigure= bypasses the repository's early Home-link preflight and is not
the supported path.

** Apply an accepted change

#+begin_src sh
ghr                               # Home-owned change
gsr                               # System-owned change
#+end_src

When both layers truly change, apply and verify the System boundary first,
including a reboot when required, then apply and verify Home.  A successful
command is activation evidence, not automatic runtime acceptance.

** Validate proportionally

Use the smallest checks that prove the edited surface.  Common live checks
include:

#+begin_src sh
scripts/emacs-check
scripts/desktop-session-check
scripts/guix-generation-retention verify
scripts/guix-generation-retention rollback-drill
#+end_src

=scripts/desktop-session-check= requires a running local EXWM session.  Add
=--require-bluetooth= only when the Bluetooth endpoint is actually available
for that acceptance pass.

The broad legacy gates, including =scripts/shell-check --strict= and
=scripts/guix-build-gate=, currently cover optional dialect fixtures and the
frozen System-pruning experiment.  They are not the default definition of
success for new work.  Use them only when their complete existing scope is
deliberately relevant; otherwise run the focused checks for the change.

** Update channels separately

The repository lock contains =guix=, =nonguix=, and =sk-guix=.  A channel
update is its own reviewed operation, not an incidental part of unrelated
configuration work.  After explicitly changing and reviewing the pinned
commits, the live pull command is:

#+begin_src sh
~/.config/guix/current/bin/guix pull -C ~/.config/guix/channels.scm
#+end_src

=scripts/guix-reconfigure= refuses activation when the active Pull profile and
repository lock disagree.

* Recovery and maintenance

=guix/machines/guixpc/generation-pins.tsv= records machine-specific accepted
System, Home, and Pull recovery targets.  The supported retention reporter has
no apply action:

#+begin_src sh
scripts/guix-generation-retention inventory
scripts/guix-generation-retention verify
scripts/guix-generation-retention plan
scripts/guix-generation-retention estimate
scripts/guix-generation-retention rollback-drill
#+end_src

Its reports do not authorize deletion and its byte counts are not guaranteed
reclaim.  System-generation deletion, bootloader reconstruction, and Guix
store garbage collection require a new, bounded plan and explicit approval.

The source checkpoint branch is not a backup of mutable documents, secrets,
keys, browser data, or other local state.  Backup and storage policy remains a
separate parked decision.

* Accepted implementation roadmap

This is the current outcome-based plan for =guixpc=.  EXWM is the permanent
desktop direction.  The target is a predictable, coherent workstation with
clear ownership, useful recovery paths, safe maintenance, a practical Scheme
learning environment, and small Lisp-native tools where they improve daily
use.  It is not a framework-adoption, language-collection, or operating-system
research plan.

Each slice is separately authorized.  C0 is closed at published checkpoint
=e8ce8b2=.  C1 is the current authorized source slice; its runtime reload and
hand test wait for a reviewed, published source checkpoint.  C2 and later work
remain unauthorized.

** C0 — Establish the human roadmap

- Result: this README and the editable Org agenda present one C0--C16 plan,
  explicit outcomes, boundaries, reference policy, and parked work.
- Boundary: documentation only.  Do not build, activate, reconfigure, reboot,
  revive P5.2b, or alter the frozen branch.  C0 closed after both documents
  were reviewed and the repository checkpoint was published.
- State: closed and published at =e8ce8b2=.

** C1 — Make EXWM input, cancellation, and launching predictable

- Result: context-aware escape/cancel behavior works across regions,
  minibuffers, transient helpers, Evil, terminals, REPLs, and EXWM; sending the
  next key to an X client remains explicit; launcher behavior is consistent
  and discoverable.  Existing Super bindings remain, while the physical
  34-key keyboard's Ctrl+Alt layer is the canonical action vocabulary and
  agrees with the macOS AeroSpace implementation.  The existing MSI/NixOS
  configuration is a partial reference whose parity gaps are recorded rather
  than copied into GuixPC.  Workspaces 6--10 are created only when first
  requested, so the accepted five-frame startup is unchanged.  The Ctrl+Alt
  aliases are deliberately EXWM-global, including inside Emacs buffers: the
  physical WM layer controls the whole session and therefore occupies those
  corresponding =C-M= editing chords.
- Reference: the exact physical source is
  =yivu-corne-v4/validation/2026-07-20-lp-34-rgb-off-left-final/stock-layout.vil=
  at content digest =afe26819eef3= (from =git hash-object=).  Cross-host
  semantics were compared with =nixos-macos/darwin/modules/aerospace.nix= at
  commit =b2ea774= and =nixos-dotfiles/config/hypr/hyprland.lua= at commit
  =73e2bd8=.  AeroSpace is the complete semantic reference.  NixOS currently
  differs for browser/Code, Shift+move, fullscreen, and reload; those are
  cross-host parity debts, not GuixPC C1 work.  Neither reference repository is
  modified by this slice.
- Boundary: hand-test the cancel/pass-through and 34-key matrices.  Do not
  rewrite completion, tabs, or the window manager; do not add launcher history,
  ranking, or a separate desktop-entry database.  C1 covers launch Terminal,
  Emacs home, Code/VSCodium, browser, files, and app picker; focus and move;
  close, fullscreen, and reload; and workspace switch/move 1--10.  Shift is
  explicit for every move action.  Float/tile, layout, and resize belong to C2,
  while clipboard and screenshot actions belong to C3.

** C2 — Normalize EXWM windows, helpers, dialogs, and recovery

- Result: Help and documentation, Dired/Ibuffer/Treemacs/Magit, compilation
  and diagnostics, warnings and backtraces, Geiser/SLY helpers, transients, X
  dialogs, fullscreen windows, startup, and winner restoration have deliberate
  placement and focus behavior.  The reserved 34-key float/tile, layout, and
  resize actions have deliberate EXWM behavior.  Add one recovery command only
  if the tests demonstrate a real need.
- Boundary: the representative window matrix, reserved-action matrix, and
  recovery hand test pass.  Do not introduce Beframe, Bufferlo, tab workspaces,
  or a master-stack rewrite.

** C3 — Complete desktop integration and attention handling

- Result: clipboard and primary selection work across Emacs, Kitty, a browser,
  and GTK; screenshots, notifications, focus requests, MIME handlers, default
  applications, and Polkit each have one understood owner.
- Boundary: test the actual applications.  Clipboard history waits for a
  privacy decision, and Qt theming waits for a real daily-use Qt application.

** C4 — Simplify and accept the existing Org workflow

- Result: capture, clarification, refile, agenda, daily review, and weekly
  review each have one obvious command and a short help surface.  Redundancy is
  removed only when it is confusing, completed-task handling is decided
  explicitly, and normal editing is free of Org lint failures.
- Boundary: Rafael completes one full representative workflow by hand.  Do not
  replace the current Org system or automatically refile DONE tasks by default.

** C5 — Give Home and XDG state explicit ownership

- Result: relevant Home paths are classified as declarative configuration,
  durable synchronized data, durable local data, secrets, logs, rebuildable
  cache, runtime state, or unknown.  One reviewed batch then moves, declares,
  quarantines, or leaves items with a rollback path.
- Boundary: no unknown file is deleted.  Inventory and approve the batch before
  changing it; do not turn hygiene into an unbounded home-directory rewrite.

** C6 — Add one Lisp-native system health view

- Result: a versioned Guile =sk-status= expression reports System, Home, and

  WirePlumber; Bluetooth; NetworkManager; relevant errors; and recovery hints.
  An Emacs view renders the same state.
- Boundary: read-only inspection only.  Do not add a daemon, database,
  telemetry collector, or a parallel source of machine truth.

** C7 — Make Git, SSH, and GPG behavior declarative and testable

- Result: Guix Home owns accepted global identity and defaults; commit-signing

  truthful =xterm-kitty= remains intact; and a read-only Guile repository audit

- Boundary: a local and SSH signed-commit test plus =emacsclient -t= pass.  No
  automatic cleaning, history rewriting, or repository mutation.

** C8 — Give Org notes private synchronization and history

- Result: a selected private remote has explicit included trees, exclusions,
  pull-before-edit and conflict rules, attachment/archive policy, and a tested
  fresh-clone, conflict, and restore workflow.
- Boundary: synchronization and version history are not described as a full
  backup.  Do not enroll secrets or unrelated mutable data implicitly.

** C9 — Rehearse security, updates, and rollback

- Result: listeners and firewall ownership, SSH policy, clock state, key

  build, activation, reboot, and rollback are rehearsed as separate stages.
- Boundary: every listener has an owner and the update rehearsal succeeds.
  The accepted trusted-residence/no-lock policy remains unless Rafael changes
  it.

** C10 — Promote only supported Guix garbage collection

- Result: the existing estimator protects current, booted-when-different, five

- Boundary: perform a second independent rehearsal before considering a
  schedule.  No System-generation deletion, custom GRUB repair, P5.2b
  dependency, or claim that estimated bytes equal reclaimed bytes.

** C11 — Make storage health, capacity, and TRIM visible

- Result: supported SMART health where safe, capacity thresholds, periodic

- Boundary: the second SSD remains unmounted and undeclared.  No repartition,

- Result: one command opens an exercise, Geiser REPL, tests, documentation, and

- Boundary: Emacs and the test runner are the terminal interface; project or

- Result: a small offline path uses Emacs Lisp Elements, fixtures, and selected

- Boundary: optional and non-blocking; it follows C12 and is not a prerequisite

** C13 — Replace Bluetooth-manager friction vertically

- Result: a Guile BlueZ D-Bus layer plus Emacs UI handles power, scan, pair,

- Boundary: retain Blueman until repeated real-device parity, recovery, and

** C14 — Add a Lisp-native sound control surface

- Result: a Guile interface to WirePlumber/PipeWire, or narrowly to =wpctl=,

** C16 — Close with a usable operator manual and bootstrap rehearsal

- Result: an indexed Org manual covers tty1, generation, EXWM, services,
  network, channel, and rollback recovery; every custom tool exposes source or
  help; a daily-use hand test and an installed-machine fresh-checkout/bootstrap
  rehearsal pass; uncovered mutable data is stated honestly.
- Boundary: documentation and proof only.  C16 introduces no new feature.

** Implementation discipline

Before a slice starts, record its objective, observable result, affected files
and live surfaces, non-goals, focused checks, build/activation/reboot and hand
test requirements, rollback, and stopping point.  Then:

1. verify the live host, checkout, declaration links, and relevant runtime;
2. make the smallest source change on =main=;
3. run focused tests for the changed surface;
4. create a bounded checkpoint for Rafael to review and publish;
5. build only the Guix layer that changed;
6. activate only after the source checkpoint is published;
7. complete runtime and human acceptance where required;
8. record the result and stop.

Stop and reassess before continuing when a second unexpected prerequisite
appears, the promised user outcome changes, or more than one correction
checkpoint is needed.  Tests and evidence support a slice; they may not become
a separate project.

** Reference policy

- [[https://github.com/abcdw/rde][RDE]] contributes principles: explicit state,
  one clear tool per job, reproducibility, offline usefulness, and
  attention-friendly behavior.  It is not adopted as a framework or desktop.
- [[https://github.com/abcdw/guix][abcdw/guix]] is an upstream Guix/API reference,
  not a new channel or dependency.
- [[https://github.com/protesilaos/dotfiles][Prot's dotfiles]] contribute contextual
  quit behavior, display-policy ideas, and simple discoverable native UI.
- [[https://github.com/protesilaos/emacs-lisp-elements][Emacs Lisp Elements]] is an optional
  offline learning reference for C12-E.
- [[https://github.com/joshuablais/studium-emacs][Studium Emacs]] contributes launcher
  and front-end ideas only.  Do not copy code without verified licensing and
  provenance, and do not import its package scope.

These references are idea catalogs, not authorities, dependencies, or implied
backlogs.  An imported idea still needs a local problem, a bounded result, and
explicit slice authorization.

* Outside the accepted roadmap

The following work is deliberately parked.  It is not a dependency for
C0--C16 and may not be used as a blocker.  The signal beside each item is what
to look for before discussing it again.

- *Leaving EXWM or adopting RDE/Sway as the desktop:* revisit only if an
  accepted daily workflow cannot be made reliable in EXWM, EXWM becomes
  unmaintainable, or a concrete security requirement cannot be met.
- *Migrating from Xorg to XLibre:* keep the packaging reference, but revisit
  only for a reproducible Xorg defect or security need and a mature,
  reviewable Guix package and rollback path.
- *General startup optimization:* revisit for repeated user-visible boot or
  EXWM latency, hangs, or interaction loss.  A benchmark opportunity alone is
  not a reason.
- *Qt normalization:* revisit when a real daily-use Qt application exists and
  visibly violates the accepted theme tokens.
- *New Lisp dialects or broad Emacs application suites:* revisit only for a
  concrete project, curriculum, or recurring workflow that the current tools
  cannot serve.
- *Large workspace, completion, tab, or launcher frameworks:* revisit only
  after a named EXWM friction survives the focused C1/C2 approach.
- *Kernel, IRQ, HPET/TSC, USB-keyboard, or other hardware experiments:* revisit
  for a repeatable disruptive symptom such as crashes, data loss, material
  latency, or device failure, with evidence that points to that layer.
- *P5.2b, D1--D5 descendants, custom GRUB work, and direct System-generation
  deletion:* remain frozen.  Revisit only if disk pressure remains after the
  supported C10 path and a simpler supported recovery route is unavailable.
- *Backup implementation, repartitioning, migration, or assigning the second
  SSD:* revisit when important local-only data grows, affordable storage
  becomes available, or Rafael's accepted risk changes.
- *RSS, mail, Ledger, Calibre, and Studium-style environment expansion:*
  revisit one application at a time when a recurring real workflow appears.
- *Replacing BlueZ, PipeWire, WirePlumber, or NetworkManager:* revisit only if
  an upstream daemon cannot meet a concrete requirement; C13--C15 build small
  Lisp-native control surfaces over the existing services.
- *A resident health daemon, database, telemetry system, or Guix framework:*
  revisit only when the read-only, on-demand C6 design demonstrably cannot
  answer an accepted operational question.

Owns normal dotfile behavior. Most changes here apply by reloading the affected

Guix Home recreates the live app symlinks to these files during =ghr=. It does
not replace them with store snapshots, so normal repo edits remain live.
=scripts/guix-reconfigure= now runs a read-only check of all 14 links through
the fixed System-profile Guile before it invokes Guix.  A regular-file,
directory, dangling-link, or missing-source conflict present at that preflight
is preserved and prevents the built-in Home activation services from starting.
The activation service repeats the same complete validation before changing any
project link.  A concurrent path race can still reach that later boundary after
earlier built-in activation, so =ghr= remains the supported path and the
additive System fallback is retained throughout this checkpoint.  Invoking
=guix home reconfigure= directly bypasses the early wrapper preflight.

Fish behavior lives in =shell/config.fish= and is installed through Guix Home
because Fish is the account login shell. Run =ghr= after editing it. The prompt
is native Fish, not Starship. Fastfetch is used as an interactive shell greeter,
so it appears in kitty or any other terminal that starts an interactive Fish
shell.  Direct SSH commands start Fish as non-login; when the Home profile is
not first in PATH, the Fish config imports =~/.profile= through the explicitly
owned =fish-foreign-env= package.  Local child Fish processes keep their
inherited environment unchanged.

This machine uses Emacs as both editor and X window manager.

** Startup path

#+begin_src text
tty1 mingetty autologin
-> Fish login shell
-> shell/config.fish guard
-> scripts/exwm-session (bounded current/previous logs)
-> startx
-> ~/.xinitrc
-> shell/xinitrc
-> emacs -fs --eval '(load-file "~/.exwm")'
-> ~/.exwm
-> emacs/exwm-loader.el
-> ~/.emacs.d/exwm.el
-> emacs/exwm.el

Regular Emacs init:
~/.emacs.d/init.el -> emacs/init.el
#+end_src

There is no graphical display manager in the intended path.  The system config
removes GDM, tty1 autologins as =skydive420dz=, and Fish starts EXWM only for
the local interactive tty1 login shell.  SSH shells are excluded by the Fish
guard.

The session runner keeps a bounded private current/previous log pair:

#+begin_src text
~/.local/state/sk-exwm/startx-current.log
~/.local/state/sk-exwm/startx-previous.log
#+end_src

If =XDG_STATE_HOME= is set, its value replaces =~/.local/state=.
The previous file appears after a legacy-log migration or the second attempt;
the state directory never grows beyond the pair.

A failed or normally ended X session returns to the existing tty1 Fish prompt
instead of terminating the login shell and triggering another mingetty
autologin.  Inspect the current log, correct the cause, and run =sk-start-exwm=
to retry.  SSH, an existing graphical display, and non-tty1 login shells remain
silent and never autostart X.

The live EXWM session should report EXWM as loaded:

#+begin_src sh
emacsclient --eval "(list :server t :daemon (daemonp) :user-init user-init-file :exwm-loaded (featurep (quote exwm)) :server-running (server-running-p))"
#+end_src

Validate the editor config with the same Guix profile assumptions used by the
live session:

#+begin_src sh
cd ~/Projects/guix-dotfiles
scripts/emacs-check
#+end_src

The checker has two independent phases:

#+begin_src text
live   one fixed read-only expression sent to the existing EXWM server
batch  copied sources under env -i with isolated HOME, TMPDIR, and every XDG path
#+end_src

The live phase never loads configuration.  It checks the running executable,
EXWM/server state, package generations, hooks, keys, and positive load-path
ownership.  The batch phase prevents writes outside its temporary root, keeps
server/history/notes startup disabled, captures warnings, runs real mode hooks
with only deferred LSP process startup stubbed, and executes ERT, formatter,
byte-compile, and Checkdoc gates.  Repository status is compared before and
after both phases.

These maintenance-only negative controls must each exit nonzero while reporting
the other phase independently:

#+begin_src sh
SK_EMACS_CHECK_BREAK_LIVE=1 scripts/emacs-check
SK_EMACS_CHECK_BREAK_BATCH=1 scripts/emacs-check
SK_EMACS_CHECK_BREAK_WARNING=1 scripts/emacs-check
#+end_src

The unmanaged Breadcrumb ELPA tree found by the audit was not used by this
configuration.  It is quarantined outside =package-user-dir= at
=~/.local/state/emacs/elpa-quarantine-20260712-slice8=.  Keep it until a normal
EXWM restart and another full check pass; then it is safe to delete.  To roll
back before deletion, move it back to =~/.emacs.d/elpa= and restart Emacs.

Do not use plain =emacs --batch= from SSH as the canonical check. Noninteractive
SSH may start with a minimal =PATH= and no =EMACSLOADPATH=, which can produce
false package warnings even when the live EXWM Emacs session can load the same
Guix-provided packages.

** Emacs language acceptance

Language support is split by backend. LSP-backed languages use =lsp-mode=,
Company, LSP UI, and =SPC c x= through the LSP diagnostics path. Non-LSP
languages use their native Emacs package plus Flycheck or REPL tooling where
appropriate. =SPC c x= dispatches by backend: LSP diagnostics for LSP buffers,
Flycheck errors for non-LSP buffers.

#+begin_src text
Lua
  Mode:        lua-mode
  Backend:     lsp-mode + lua-language-server
  Diagnostics: LSP/Flycheck path
  Formatter:   native Emacs indentation through =SPC c f=
  Notes:       LSP-backed language model baseline. Stylua is intentionally not
               installed because it is not packaged in the current Guix channel
               and would require local Rust crate packaging.

Python
  Mode:        python-mode
  Backend:     lsp-mode + python-lsp-server / pylsp
  Diagnostics: LSP/Flycheck path
  Formatter:   ruff format through =SPC c f=
  Notes:       Pyright is not required for the current baseline. Formatting is
               manual only; there is no format-on-save policy.

Shell
  Mode:        sh-mode
  Backend:     Flycheck sh-bash + shellcheck
  Diagnostics: Flycheck path
  Notes:       shfmt is installed for formatting, but no format-on-save policy.

Lisp-family editing has a shared leader surface over the native backend for
each dialect:

#+begin_src text

SPC l b  evaluate buffer
SPC l d  evaluate defun / top-level form
SPC l e  evaluate last sexp
SPC l k  show docs at point
#+end_src

These are wrappers over existing commands, not replacements for native Emacs
keys:

#+begin_src text

Cold Lisp files should provide mode detection, indentation, basic syntax, and
the correct command surface. Rich semantic completion/docs are REPL-backed for

#+begin_src text

Formatter decisions are intentionally conservative:

#+begin_src text

        .clang-format discovery.

** Compositor

Picom is intentionally scoped to Emacs frame transparency only:

#+begin_src text
picom --backend glx --vsync --opacity-rule '85:class_g = "Emacs"'
#+end_src

The X session starts the wallpaper and Picom before launching Emacs/EXWM so the
compositor is already active when Emacs appears. It launches Emacs fullscreen
directly instead of using the Guix =exwm= wrapper, whose hardcoded =-mm=
maximized-frame startup can briefly draw a small pre-maximized frame.  Xinit is
the only automatic wallpaper/Picom owner; EXWM startup and reload never repeat
either operation.  =M-x sk/start-picom= and =M-x sk/set-wallpaper= remain
explicit maintenance commands.  Browser and terminal transparency are not
managed through global active/inactive opacity rules.

** Application placement and reload policy

Desktop entries are fully resolved and validated before a process starts.  The
launcher omits =Hidden=/=NoDisplay= entries and conservatively omits complex
=Exec= quoting that the direct parser cannot reproduce faithfully.  It never
creates a speculative split.  Instead, a 30-second launch intent records the
origin workspace, process PID, and normalized class metadata.  The EXWM manage
hook rejects transient, dialog, fixed-size, and floating clients; it prefers
exact/descendant PID matching and allows live class fallback only for the known
single-instance Chromium path.  An ambiguous match is ignored.  Failure,
timeout, successful placement, and reload all remove the intent and timer.

Configuration reload reader-checks every module before loading the first one.
It reports the exact failed module and how many earlier modules loaded, restores
the complete project display-policy state, and explicitly warns that arbitrary
earlier function/hook mutations cannot be rolled back.  Project display rules
are identity-owned and preserve foreign/package rules.  The reviewed runtime
floor is EXWM 0.35; its remaining private workspace/floating adapters are
guarded by that exact compatibility check.  The Xref RET adapter is reviewed on
Emacs 30 and degrades with a warning instead of blocking editor startup on a
future Emacs major release.

Live acceptance for the display/session path:

#+begin_src text
Xorg initial mode: 3440x1440_100
xrandr current:    3440x1440_100 100.00*+
Emacs frame:       3440x1440 fullboth
#+end_src

Do not preload =amdgpu= through =initrd-modules= on this machine. That experiment
made X fall back to a generic =default= 1024x768 screen. Reverting it restored
the accepted =HDMI-A-0= / =3440x1440_100= path.

=s-h= uses the normal left-window focus path first. If normal windmove cannot
see a left target and Treemacs is open, it falls back to selecting Treemacs.

Workspace numbers are user-facing and one-based. EXWM internally uses zero-based
workspace indexes, so =s-1= maps to EXWM workspace index 0, =s-2= maps to index
1, and so on.

** Future project: Lisp-native user environment

Long-term direction: make GuixPC a practical Lisp-centered workstation where the
daily user experience is owned by Emacs/EXWM, Emacs Lisp, and Guile instead of a
collection of separate GTK/Qt desktop tools.

This does not mean replacing the kernel, drivers, browser, or core daemons. The
intended boundary is:

#+begin_src text
Core system:
  Linux-libre or selected nonguix kernel
  Guix / Shepherd
  Xorg / EXWM
  D-Bus
  NetworkManager
  PipeWire / WirePlumber
  BlueZ

Native user environment:
  Emacs / EXWM
  Emacs Lisp UI panels
  Guile system helpers where appropriate
  Org/Dired/Eshell/Vterm/Treemacs/Magit as primary workflow surfaces

External exception:
  browser
  unavoidable hardware/vendor tools
#+end_src

The goal is to reduce daily dependence on standalone desktop applications, not
to remove every non-Lisp binary from the system closure. Chromium, Emacs builds,
Xorg, graphics/audio/network services, and hardware support will still pull
normal system libraries.

Nyxt was evaluated as a possible programmable Lisp browser. The concept matches
this machine's direction, but it is paused for now:

- official Guix does not currently provide =nyxt=;

** EXWM app launcher

=s-SPC= opens a Counsel/Ivy app launcher from installed desktop entries.

The first implementation tried to delegate launching to =gtk-launch=, matching
what =counsel-linux-app= does upstream. On this Guix system neither
=gtk-launch= nor =gtk4-launch= is present in the active profile, so the local
launcher keeps those as optional fast paths and otherwise launches directly from
the selected =.desktop= file:

1. resolve the selected desktop id, for example =blueman-manager.desktop=
2. find the desktop file under the application directories
3. read the =Exec== field from the =[Desktop Entry]= section
4. remove desktop-entry field codes such as =%U=, =%F=, and =%i=
5. launch the command asynchronously with Emacs =start-process=

Terminal desktop entries are launched through =kitty -- ...=. This keeps GUI
app launching usable without adding another standalone launcher while preserving
the documented desktop-entry model.

Reference points:

- Desktop Entry Specification, =Exec= field codes:
  https://specifications.freedesktop.org/desktop-entry-spec/latest/exec-variables.html
- Emacs asynchronous process launch:
  https://www.gnu.org/software/emacs/manual/html_node/elisp/Asynchronous-Processes.html

Bluetooth Manager can be tested through the launcher by selecting
=Bluetooth Manager=, or directly with:

#+begin_src sh
blueman-manager
#+end_src

If Blueman opens with icon warnings, that is a GTK/icon-theme issue, not the
app launcher path.

* Desktop session policy

The accepted Slice 10 baseline gives each long-lived desktop service one owner:

#+begin_src text
session D-Bus             Guix Home at unix:path=/run/user/1000/bus
PipeWire/WirePlumber      Guix Home Shepherd services on that bus
realtime scheduling       system RTKit through D-Bus/Polkit
EXWM and launched apps    inherit the Guix Home bus directly
AT-SPI accessibility bus  separate purpose-specific bus under /run/user/1000/at-spi
#+end_src

=shell/xinitrc= refuses an absent or noncanonical Home bus instead of silently
autolaunching a second one.  It updates only the graphical D-Bus activation
variables, without =--all= or systemd integration, then starts Emacs directly.

The machine never suspends itself through elogind.  X independently blanks and
powers down the monitor after 600 seconds of X input inactivity, with standby,
suspend, and off all set to 600 seconds.  This preserves the observed desktop
behavior and protects the panel; it is not a screen lock.  Tty1 autologin
remains a separate Slice 19 trust decision.

The graphical login is intentionally still represented by elogind as the
active local =tty1= session with =Type=tty= and a tty-derived idle hint.  EXWM
children receive correct =DISPLAY=:0=, =XDG_SESSION_TYPE=x11=, and desktop
metadata.  Correcting login1's type would require changing how the PAM session
is created, not an Xinit environment tweak, and is outside this slice.

After a fresh system/session activation, validate the automated boundary with:

#+begin_src sh
scripts/desktop-session-check
#+end_src

The stricter Bluetooth gate expects the paired Bose speaker to be powered and
connected:

#+begin_src sh
scripts/desktop-session-check --require-bluetooth
#+end_src

The accepted run used the strict Bluetooth gate with the Bose Mini II as the
default sink, then manually confirmed audible Bluetooth output and both XF86
volume keys.  A real Kitty launch through EXWM's desktop-entry fallback inherited
the canonical bus.  The HDMI endpoint and targeted automated stream remain
healthy, but audible HDMI output is intentionally outside this machine's
acceptance policy because no HDMI audio device is used.

* Guix channels

The version-controlled =guix/channels.scm= is the channel lock.  Every channel
has an exact 40-hex commit and authenticated introduction; moving branches are
metadata only and never select a revision by themselves.

#+begin_src text
guix     upstream GNU Guix packages and system modules
nonguix  nonfree kernel/firmware transformation and substitutes
sk-guix  personal package channel; only promoted top-level package is kitty-latest
#+end_src

The prepared Slice 12 lock is:

#+begin_src text
guix     a8391f2d7451c2463ba253ffa9872fa6f27485d7
nonguix  3b66965566fe8c96edb5a41fd39a9e5a90ad9b61
sk-guix  b84ff24a515b2b6d0db85c0da30960c96e20ad22
#+end_src

=sk-guix= authentication starts at =eaade3680892ce74ebae68f4922cd0eb4a463a17=
with signer =6B09 4D15 B02E 54B0 F6B5 E9E8 6F83 FC62 D232 E5EC=.  The public
key is on its orphan =keyring= branch.  Main commits from the introduction
forward must remain signed and authorized; replace/authorize the current key
before its 2028-07-11 expiry.

=sk-guix= owns =kitty-latest= only. The system imports it through =(sk packages
terminals)= and installs =kitty-latest= directly, instead of resolving =kitty=
through =specification->package=. Other applications should stay on upstream
Guix/Nonguix unless we explicitly decide to package and maintain them in
=sk-guix=.  =fish-latest= remains quarantined: the gate proves its derivation
evaluates, but does not promote or build the known-incomplete recipe.

Before changing the active pull profile, run the non-activating gate:

#+begin_src sh
scripts/guix-build-gate --plan
scripts/guix-build-gate
#+end_src

It creates isolated HOME, cache, and trusted-channel state, realizes the exact
authenticated time-machine profile, verifies provenance and reviewed EXWM 0.35,
builds Kitty and its maintained Go inputs, evaluates quarantined Fish, builds
each development manifest independently, then builds Home and System.  Full
mode also enters each pure manifest for isolated runtime and cross-tier leakage
probes and validates the realized System/Home ownership boundary and critical
profile artifacts; plan mode remains evaluation-only.  Guix may reuse already-authenticated
immutable store objects; the separate isolated clean-clone test verifies the
published =sk-guix= refs.  The gate never pulls, reconfigures,
switches/deletes generations, or runs garbage collection.

To update channels, change only the recorded commits while retaining each
introduction, review the diff, and pass the complete gate.  Promote the tested
lock explicitly:

#+begin_src sh
~/.config/guix/current/bin/guix pull -C ~/.config/guix/channels.scm
~/.config/guix/current/bin/guix describe --format=recutils
gsr
ghr
#+end_src

=scripts/guix-reconfigure= refuses both =gsr= and =ghr= when the active pull
profile differs from the repository lock.  If a new pull frontend regresses,
switch to the exact protected pull generation before rebuilding from the old
lock; do not use a relative roll-back when the intended generation is known.

* Generation retention and rollback

Machine-specific keep targets live in
=guix/machines/guixpc/generation-pins.tsv=.  Each row binds a kind and
generation number to its expected canonical store target and recovery role.
The planner aborts on pin/target mismatches, current/runtime divergence, an
unrooted booted System, or dangling targets.  The rollback drill additionally
checks recovery artifacts and required GRUB generation references.

#+begin_src sh
scripts/guix-generation-retention inventory
scripts/guix-generation-retention verify
scripts/guix-generation-retention plan
scripts/guix-generation-retention rollback-drill
#+end_src

The current implementation is intentionally read-only.  It reports only older
generation links whose identical canonical closure is retained by a newer
generation; it has no =apply= action and never invokes =guix gc=.  Unique
closures are not future deletion candidates unless they are older than 30 days,
outside the newest three distinct closures, unpinned, not current/booted, and
superseded by an accepted realization.  Any mutating cleanup needs a separate,
reviewed implementation and a freshly generated exact-number plan.  The
read-only tool currently proves retention of the booted closure; an apply
implementation must additionally identify and protect the exact booted
generation-number link before deleting anything.

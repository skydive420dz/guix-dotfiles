#+title: P4.1 recovery inventory and backup decision packet
#+date: 2026-07-18
#+status: complete at metadata/source boundary; backup policy and restore remain open

* Outcome

P4.1 establishes the first durable recovery inventory for GuixPC without
selecting a backup product or destination.  It records what is declarative,
what is held off-machine through Git, what exists only on the internal SSD,
which state is sensitive, and which choices must be accepted before P4.2 can
write a backup.

The inventory commands read filesystem, repository, key, agent, generation,
mount, and service metadata.  They did not read private-key contents, browser
preferences, Org-note contents outside the in-scope roadmap, shell histories,
or credential payloads.  The repository documents and editable roadmap were
copied through temporary workspaces for authorized editing and linting; that
staging is not a durable or restore-tested backup.  The slice added no package
or service, wrote no backup payload or secret, changed no permission, switched
no generation, activated no configuration, restarted no process, and performed
no reboot.

One command violated the intended read-only boundary in a narrow, confirmed
way: =guix gc --list-dead= pruned 26 already-dangling automatic GC-root
symlinks while enumerating dead store items.  No store object or generation
was deleted.  The exact side effect and after-state are recorded below so it
does not disappear behind a false “read-only” claim.

The source correction in this slice updates the Home recovery-role metadata to
the already accepted P3.4 state: Home 47 is the promotion rollback and Home 46
is the preferred rollback.  Home 42 and 41 remain retained older
=last-known-good= records.  This changes no live profile or generation link.

* Selection and stop boundary

Phase 4 is the highest-priority open roadmap phase.  Its only declared
prerequisite is the completed Phase 0 baseline.  Phase 5 cleanup and garbage
collection remain explicitly blocked by a restore-tested backup.

P4.1 is limited to:

1. current identity, storage, generation, and repository state;
2. location/size/count/owner/mode metadata for recovery-relevant paths;
3. SSH/GPG public identity, expiry, permission, and agent metadata;
4. a coverage and restore-order matrix;
5. confirmed gaps and the exact decisions required for P4.2; and
6. a source-only correction where existing accepted recovery roles and the pin
   file disagree.

P4.1 does not choose a destination, encryption method, backup tool, cadence,
retention window, recovery-point objective, recovery-time objective, or
failure-notification channel.  It does not claim backup or restore acceptance.

* Recovery dependency order

A complete recovery cannot begin with =guix-bootstrap= alone.  The current
safe order is:

1. bootable installer/recovery media, partitioning, filesystems, EFI/GRUB,
   firmware/network, and the user account;
2. trusted network and Git access, including SSH material if private access is
   required;
3. Guix channels and both Git repositories;
4. the System configuration and a bootable System generation;
5. the matching repository checkpoint before any Home generation rollback or
   activation;
6. GPG signing/trust state before new authenticated channel history;
7. Org and other irreplaceable user data; and
8. selected browser/editor/application state.

The existing bootstrap and retention tooling cover important installed-system
boundaries, but not steps 1, 2, 6, 7, or 8 as a blank-disk recovery workflow.

* Unexpected GC-root side effect

The command was:

#+begin_src sh
~/.config/guix/current/bin/guix gc --list-dead
#+end_src

While producing the list, Guix reported removing 26 stale symlinks under
=/var/guix/gcroots/auto=.  Twenty-four targeted already-absent per-user
profile roots; two targeted already-removed temporary
=guix-bootstrap-acceptance= trees.

Afterward:

- =/var/guix/gcroots/auto= contains 44 symlinks;
- a filesystem-level dangling-link count is zero;
- System/Home/Pull link counts remain 86/47/5;
- current and booted System remain generation 87;
- current Home remains generation 47;
- current Pull remains generation 5; and
- =scripts/guix-generation-retention verify= still passes.

No =guix gc --delete=, unqualified collecting =guix gc=, generation deletion,
switch, reconfigure, or activation command ran.  Command output reported no
store-object or generation deletion.  Recreating links whose targets no longer
exist would restore stale metadata rather than a usable root, so no
counter-mutation was attempted.

Future no-mutation reporters must use direct filesystem/profile metadata or an
isolated fixture.  They must not describe =guix gc --list-dead= as read-only.

* P4.2 decision packet

The next slice cannot select policy without Rafael's input.  It needs:

1. mandatory data:
   - confirm Org notes;
   - confirm SSH/GPG;
   - decide Chromium;
   - decide Downloads;
   - identify any other irreplaceable path;
2. one off-machine destination with known usable capacity;
3. encryption policy and a recovery credential stored independently of this
   SSD;

8. the representative files and secret-safe checks for a disposable restore
   drill.

Tool selection follows those answers.  Installing a convenient tool first
would reverse the dependency order.

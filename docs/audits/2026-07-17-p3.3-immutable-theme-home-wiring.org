* Immutable build architecture

The new =(sk theme-home)= module constructs one =computed-file= bundle.  Its
only explicit inputs are Guix-captured =theme/tokens.scm= and the accepted
wallpaper.  The derivation imports =(sk theme)=, requires production-kind
tokens, validates the copied asset below its explicit output root, and renders
the canonical six targets inside the isolated build environment.

#+begin_example
adapters/emacs.el
adapters/kitty.conf
adapters/fish.fish
adapters/gtk3.ini
adapters/gtk4.ini
adapters/x-session.sh
tokens.scm
assets/wallpapers/waifu-cyberpunk.png
#+end_example

It does not read =$HOME=, the live checkout, profile, display, process state,
dconf, fontconfig state, or the network while rendering.

Guix Home maps the bundle to:

| Home target | Bundle member |
|-------------+---------------|
| =~/.config/emacs/sk-theme-generated.el= | =adapters/emacs.el= |
| =~/.config/kitty/kitty.conf= | =adapters/kitty.conf= |
| =~/.config/gtk-3.0/settings.ini= | =adapters/gtk3.ini= |
| =~/.config/gtk-4.0/settings.ini= | =adapters/gtk4.ini= |
| =~/.config/sk-theme/x-session.sh= | =adapters/x-session.sh= |
| =~/.config/sk-theme/tokens.scm= | =tokens.scm= |
| =~/.local/share/sk-theme/assets/wallpapers/waifu-cyberpunk.png= | immutable wallpaper |

* Atomic Kitty ownership transition

=~/.config/kitty/kitty.conf= was removed from the live repository-link list in
the same source change that adds its complete Home XDG file.  It is retained in
a one-entry retired-transition manifest.

The candidate activation script orders Guix Home's =update-symlinks= before
the custom =sk:activate-repo-links= action.  On P3.4 activation, Home therefore
backs up the old repository symlink, installs the store-backed generated file,
and only then refreshes the remaining repository links.  This is one
generation-level ownership migration; Guix still briefly deletes and recreates
the individual symlink and does not promise a syscall-level atomic rename.

The read-only =ghr= preflight accepts only:

1. a missing retiring target;
2. the exact old repository symlink; or
3. a symlink resolving below =/gnu/store= whose bytes equal the reviewed
   production Kitty golden.

Regular files, wrong links, dangling links, mutable lookalikes, and
store-backed byte drift fail before reconfiguration.  The legacy
=kitty/kitty.conf= remains tracked because generation 42's stored activation
manifest still needs it for complete rollback.

* Consumer transition behavior

Emacs loads the generated adapter before =use-package=, =sk-ui=, and Org only
when the configured file resolves below =/gnu/store=.  A missing or mutable
lookalike is ignored.  Before P3.4, =sk-ui= therefore preserves the exact
Iosevka/Modus behavior.  Once the generated feature is loaded, the legacy
theme/font setup is skipped and Org inherits the generated fixed-pitch role.

Xinit applies the same rule to the generated same-shell adapter.  Until it is
active, the existing wallpaper, zoom fit, Picom backend/VSync, 85-percent Emacs
opacity, and no cursor override remain unchanged.  With the immutable adapter,
Xinit resolves the wallpaper through =XDG_DATA_HOME=, enumerates the accepted
fit and Picom values, and exports only the cursor identity/size.  It does not
force GTK, GDK, Qt, DPI, or scale environment variables.

Fish source no longer owns raw prompt colors.  The active generation still has
its old immutable file; only the later candidate activation will select the
generated prompt, syntax, autosuggestion, search, and pager policy.

* Corrections found by the real build

The dry-run evaluated successfully and predicted only Papirus and Bibata as
new downloads.  The first real derivation then exposed two module-closure
errors:

1. =(guix build utils)= was used by the builder but omitted from the imported
   module closure;
2. =source-module-closure='s default selector includes Guix modules but excludes
   the project-local =(sk theme)= module.

The final module imports Guix's build closure and explicitly admits
=(sk theme)= through a narrow selector.  The resulting build compiled both
modules inside the derivation and succeeded.

Candidate inspection then showed that =home-fish-configuration='s =aliases=
field serializes after all =config= fragments.  Although the generated theme
was semantically effective, it was not the final auditable fragment.  The
aliases remain Scheme-owned but now occupy an explicit fragment before the
theme adapter.

These were candidate-build corrections.  None reached the active Home.

* Rollback and P3.4 handoff

P3.3 rollback before activation is a source revert only.  The realized
candidate may be garbage-collected without affecting generation 42.

P3.4 must begin only after the P3.3 source checkpoint is pushed and upstream
equality is verified.  Its gates are deliberately sequential:

1. record the exact source checkpoint, clean ahead/behind evidence, previous
   Home generation 42/result, retained =kitty/kitty.conf=, and rollback plan;
2. run the read-only 14-live-plus-one-retired link preflight;
3. run =ghr= and record the new Home generation/result plus the standard
   legacy-backup path; this proves activation only;
4. before restarting consumers, prove every generated target is store-backed
   and golden-equal, Kitty has exactly one Home owner, and all 14 continuing
   repository links remain correct;
5. test fresh Kitty, Fish, Emacs, one GTK 3 application, and
   =gtk4-widget-factory= separately;
6. because Xinit owns the session adapter, restart X/EXWM or reboot only as a
   later recorded gate, then inspect Papirus/hicolor fallback, Bibata cursor,
   Picom, wallpaper fit, transparency, and the real ultrawide display; and
7. obtain Rafael's real-display acceptance and recheck adjacent SSH/EXWM
   behavior.

Stop and roll back on preflight failure, dual or mutable Kitty ownership,
non-store/generated byte drift, parser/load regression, session instability,
or unacceptable contrast/appearance.  Preserve both halves of Kitty rollback:
generation 42 and its retained repository source.  Do not claim Qt, forced
scaling, or global toolkit environment ownership.

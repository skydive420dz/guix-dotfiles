* Recovery-root lifecycle

The fixture uses direct synthetic roots beneath:

#+begin_example
/var/guix/gcroots/p52b-system-prune/MANIFEST-SHA/
#+end_example

The closed manifest requires:

- one exact, directly named recovery root for each selected generation;
- one root for the old =bootcfg= store item; and
- one root for the new =bootcfg= store item.

For the two-generation fixture this is four roots.  Each root is a manifest-
named symlink to an exact synthetic store target.  The engine creates and
directory-syncs every root before changing GRUB, =bootcfg=, or a generation
link, and rechecks the complete root set immediately before each selected-link
move or discard.  Expected missing subsets can be recreated during pre-commit
recovery only from the closed manifest and existing exact targets.  A
retargeted, occupied, unexpected, or otherwise foreign root state fails
closed.

These paths model the required production lifecycle; they are not registered
with the live Guix daemon and are not evidence that the future live root
backend is ready.

* Durable journal

The transaction directory is keyed by the complete manifest SHA256 beneath:

#+begin_example
/var/guix/profiles/.p52b-system-prune-transactions/
#+end_example

Its journal header binds the journal schema, manifest hash, fixture-only mode,
and transaction identity.  Every event contains:

1. a strictly increasing sequence number;
2. a closed event name;
3. a manifest-constrained subject;
4. the preceding record digest; and
5. the SHA256 of the current payload.

The chain provides deterministic integrity checking for this fixture.  It is
not authentication against a same-UID actor who can rewrite and rehash the
whole journal.

The complete journal is replaced through Guix's synchronized atomic-file
helper.  The candidate additionally =fsync=s the journal directory after each
rename.  Newly staged GRUB bytes are written with =O_EXCL=, =fdatasync=, exact
mode, and a parent-directory =fsync=; GRUB and =bootcfg= promotion then use
same-directory atomic renames followed by directory =fsync=.

The old installed GRUB bytes are copied into the transaction directory,
durably written, and checked against the manifest before the forward path
continues.  Unknown journal rows, broken hash chains, reordered or duplicated
sequences, wrong subjects, truncated state, or orphaned atomic-journal
temporaries require review rather than inference.

* Interruption scope

The declared matrix injects process termination only at 98 named boundaries
after the preceding atomic or synchronized operation has returned.  It does
not inject failure inside =mkdir=, write, =fdatasync=, =fsync=, rename, or
journal replacement, and it is not a power-loss test.

In particular, D3a deliberately fails closed for a transaction directory or
quarantine left before durable =BEGIN=, a partial direct backup, and an
orphaned atomic-journal temporary.  Those states require manual review; they
are not automatically classified as rollback.  D4 must add and fixture-test
bootstrap and known-temporary reconciliation before any live promotion.  The
D3a result therefore supports the exact declared restart boundaries only, not
arbitrary-instruction or power-loss recovery.

* Forward order

The checkpoint deliberately applies the safe order:

1. validate the exact prestate under both cooperative locks;
2. durably write and verify the old installed-GRUB backup;
3. create and verify every recovery root;
4. atomically install the retained-only GRUB bytes while all links still
   exist;
5. atomically promote the new =bootcfg= root;
6. move only the selected generation symlinks into quarantine;
7. discard the quarantined symlinks only while recovery roots remain exact;
8. verify the complete retained-link/GRUB/=bootcfg=/protected-surface
   poststate;
9. append and sync =COMMITTED=;
10. remove recovery roots; and
11. append and sync =COMPLETE=.

Installing the retained-only menu before link exclusion prevents a committed
or interrupted state from advertising already removed candidate links.

* Recovery decision

Recovery is determined by the durable =COMMITTED= boundary:

| Journal boundary | Recovery direction |
|------------------+--------------------|
| No =COMMITTED= event | Ensure exact roots, restore selected links, old GRUB, and old =bootcfg=; verify prestate; remove roots; record =ROLLED-BACK= |
| =COMMITTED= present | Preserve the accepted forward state, finish remaining root cleanup, verify poststate, and record =COMPLETE= |
| Terminal =ROLLED-BACK= or =COMPLETE= | Verify the exact terminal state and return a no-op without changing the journal |

Pre-commit rollback restores link tuples before restoring old GRUB and old
=bootcfg=.  An occupied live link, ambiguous live/quarantine pair, missing
backup, unknown GRUB bytes, unknown =bootcfg= tuple, or foreign root stops
recovery without deleting evidence.

* Cooperative exclusion

The fixture acquires two non-blocking advisory locks in a fixed order:

1. the transaction-wide lock beneath the transaction base; then
2. the synthetic =system.lock= beside the System profile.

Each is a persistent, non-following regular-file lock whose inode is retained
across actions; unsafe links, shared inodes, foreign contents, or writable
parent directories fail closed.  Contention is a hard failure.  This proves
cooperation among instances of the fixture engine.  It does not exclude
arbitrary =guix system= processes:
pinned =guix system= paths do not honor this lock.  A later production design
must preserve the cooperative lock, add attended quiescence and exact
phase-by-phase revalidation, and must not describe the lock alone as global
System-profile exclusion.

* Explicit non-claims

D3a does not claim:

- arbitrary-instruction, mid-system-call, or power-loss recovery;
- automatic classification or cleanup of pre-=BEGIN=, partial-backup, or
  orphaned atomic-temporary state;
- hostile same-UID mutation resistance or immutable identity between source
  hash preflight and Guile loading;
- production GC-root registration or liveness;
- exclusion of arbitrary =guix system= processes;
- a live deletion manifest or authorization;
- installation of the D2b artifact;
- any live System-link, GRUB, =bootcfg=, EFI, profile, or store mutation;
- GC, dead/live enumeration, collection, activation, reconfiguration, or
  reboot; or
- D4 or P5.3 readiness.

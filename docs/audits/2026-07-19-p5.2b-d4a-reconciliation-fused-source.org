decision D-41.  The candidate adds:

- one closed hash-chained journal codec and legal-history automaton;
- a dependency-injected root backend and central orchestration/pre-phase gate;

hard-refuses root =/= and performs no production root registration, Guix store

* Implemented architecture

** Journal boundary and legal automaton

=(sk system-pruning-boundary)= owns canonical event encoding, sequence and
previous-hash validation, hash-chain verification, legal successors, terminal
exclusion, and the closed forward/rollback/post-commit history automaton.
Hash-valid histories with duplicate, skipped, reordered, direction-changing,
or post-terminal events fail before a phase can start.  Public trace data is
detached from the private dynamic cache, and fixtures prove mutation
isolation.

** Root backend and orchestrator

=(sk system-pruning-root-backend)= defines the one-session temporary/direct-
root, registration, validity, and liveness contract.  D4a supplies only a
synthetic backend.  =(sk system-pruning-orchestrator)= validates exact root
tuples and cleanup order and funnels every side-effecting fixture phase through
one central pre-phase gate.

The gate requires the expected journal/root/protected-surface state, attended
quiescence observation, no overlapping phase, and zero active effect before it
allows a fixture thunk.  This retains the cooperative transaction-then-System
lock order without claiming exclusion of arbitrary =guix system= processes.

** Reconciliation

=(sk system-pruning-reconciliation)= classifies only the closed accepted
synthetic filesystem states.  Exact absent, complete, provable-prefix backup,
known temporary, and bootstrap-prefix states have deterministic successors.
Foreign, multiple, mistyped, retargeted, incomplete, or ambiguous states
return =REVIEW-REQUIRED= without cleanup.

** Phase engine and in-memory runtime

=(sk system-pruning-phase-engine)= derives the closed forward, rollback,
restart, and terminal-cleanup phase registry from the validated manifest and
root tuples.  =(sk system-pruning-fixture-runtime)= executes that registry
against an in-memory model while counting every effect, phase, session, root,
journal, and protected-surface invariant.

The fused driver keeps three action families distinct:

- =fixture-points= / =fixture-verify= / =fixture-apply= /
  =fixture-recover=: the D4a in-memory phase runtime;
- =fixture-reconcile=: the synthetic filesystem reconciler; and
- explicit =legacy-fixture-*= actions: the retained D3 transaction oracle.

The D4a runtime reports
=adapter=IN-MEMORY root-role=READ-ONLY-ELIGIBILITY
supplied-root-transaction-mutations=0=.

** Fused renderer and loader

=(sk system-pruning-fused-source)= is a pure deterministic renderer.  It
accepts a closed association list, verifies every manifest-bound input, and
emits one Scheme source containing ten ordered packets:

1. root backend;
2. boundary;
3. orchestrator;
4. reconciliation;
5. embedded context;
6. transaction core;
7. phase engine;
8. fixture runtime;
9. generated embedded inputs; and
10. inert fused driver.

Each packet contains its label, exact module name, SHA256, UTF-8 size, and
source string.  At program startup, inherited Guile paths are replaced with
the pinned immutable closure, auto-compilation is disabled in the shebang, and
the canonical root-owned unwritable =/gnu/store= program location is proved
before any project form is parsed or evaluated.

One preflight validates all ten packet shapes, order, hashes, sizes, exact
module declarations, complete reader consumption, and absence of same-name
modules.  Only after every packet passes does one evaluator create each module
in dependency order, prove exact registry identity, evaluate remaining forms
in that explicit module, and restore the caller module.  The inert driver
exports =sk:main=; the rendered tail invokes it exactly once and last.

No runtime path loads or opens project source, the checkout manifest, crash
registry, retained GRUB, or legacy fixture inputs.  Those exact bytes are
already data inside the rendered source.

* Loader integration correction

The first exact rendered-source compile exposed a real integration defect.
The renderer had concatenated ten independent =define-module= sources into one
compiler lexical unit.  Guile then correctly emitted warning-fatal shadowing
and wrong-arity diagnostics even though every source compiled independently.

The correction preserves each reviewed source as data and loads it into its
own explicit module only after the all-packet preflight.  This removed the
lexical collision without weakening packet identity, compile policy, or
runtime repository isolation.  A second focused probe also showed that a
hostile =GUILE_AUTO_COMPILE=1= can compile the outer script before in-source
sanitization; the exact immutable shebang now supplies =--no-auto-compile= so
that pre-startup cache write cannot occur.

The corrected exact rendered source compiles with =guild compile -W3= and
zero diagnostics.  A mutable executable copy with hostile source, compiled,
extension, and auto-compile settings refuses at the canonical store-location
guard before a project module or poison sentinel can run.

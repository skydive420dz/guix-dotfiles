* First root cause: GTK output exposure

Generation 43 reports only:

#+begin_example
gtk 4.22.1 bin /gnu/store/hnf5in75v83zmhcqxpbflvizdbcp8a2n-gtk-4.22.1-bin
#+end_example

That output references:

#+begin_example
/gnu/store/qk4czza15lsr1d1ir6xni5613knqqz49-gtk-4.22.1
#+end_example

The referenced GTK =out= store item contains the ColorChooser XML and its own
compiled cache.  A closure reference keeps those bytes alive, but does not link
them into the Home profile.  Generation 43's profile therefore lacks the XML,
and its existing profile-generated =gschemas.compiled= does not register the
ColorChooser schema.

The correction is package ownership, not a GTK patch or environment override.
It does not add =glib:bin=, copy schema files manually, or globally set
=GSETTINGS_SCHEMA_DIR=.

* Second root cause: GStreamer plugin exposure

Generation 44 proves the schema correction, but its exact-profile factory
launch aborts with status 133, signal 5:

#+begin_example
GStreamer-Play-ERROR: GstPlay: 'playbin3' element not found
#+end_example

The active Home profile has no top-level =gst-plugins-base= or
=gst-plugins-good= record and no corresponding playback plugin links.  The
running EXWM Emacs inherited an obsolete =GST_PLUGIN_PATH= naming a nonexistent
=~/.guix-profile/lib/gstreamer-1.0= directory and has no
=GST_PLUGIN_SYSTEM_PATH=.  GTK's closure references keep GStreamer store items
alive, but do not expose their plugins through the Home profile or its search
path.

The factory embeds a VP9-in-WebM video.  =gst-plugins-base= supplies
=playbin3= through =libgstplayback.so=; =gst-plugins-good= supplies the
Matroska/WebM demuxer, VP9 decoder, =scaletempo=, and automatic audio/video
sinks.  A process-local base-only diagnostic kept the factory alive but still
emitted GStreamer errors.  A disposable base-plus-good profile kept it alive
without GStreamer errors; only GTK layout warnings remained because EXWM tiled
the deliberately oversized factory into 896 pixels while it requested at
least 1445.  Those override-based diagnostics prove the package set but are
not active-profile or visual acceptance.

The second correction directly selects both normal default-output packages.
It does not hard-code =GST_PLUGIN_PATH=, =GST_PLUGIN_SYSTEM_PATH=, or a user
registry.

* Fourth root cause: first-frame ordering

=scripts/exwm-session= runs Xinit, and =shell/xinitrc= applies the generated X
policy, wallpaper, and Picom before:

#+begin_example
exec emacs -fs --eval '(load-file "~/.exwm")'
#+end_example

Emacs loads =early-init.el= before constructing the initial graphical frame,
but the active file owns only startup tracing and native-comp cache selection.
=init.el= loads the generated adapter after the initial frame exists, then
loads =sk-ui=.  Only =sk-ui= disables the menu, tool, scroll, and tooltip
modes and sets fringes.  Those are confirmed late redraw/geometry owners.

The later =~/.exwm= load starts EXWM after normal init.  EXWM creates four
additional workspace frames and defers their fullscreen transition until idle.
The active initial frame receives fullscreen from =-fs=, but
=default-frame-alist= has neither fullscreen nor undecorated defaults.  All
five final live frames reported =undecorated=nil=.  This is confirmed
construction ordering; its attribution to the exact titlebar pixels is a
strong source hypothesis because no startup video trace exists.

No current causal module ranking is claimed.  The observed
=emacs-init-time= was about 8.99 seconds, but startup attribution was not
enabled on that boot.  The finding concerns when visible state is applied, not
proof that the same code dominates total startup time.

The correction:

- installs menu/tool/scrollbar, fringe, fullscreen, and undecorated parameters
  in both =initial-frame-alist= and =default-frame-alist= during early init;
- inhibits the startup screen/message at that same pre-frame boundary;
- consumes the immutable generated theme in =early-init.el= after native-cache
  redirection and removes the late =init.el= owner;
- keeps GUI-dependent global mode calls in =sk-ui= after GUI initialization;
  and
- changes the generated symbol-font block into an idempotent per-frame helper
  registered in =after-make-frame-functions= from early init, then applies it
  to any already-existing graphical frames.

The per-frame path is necessary because a pre-frame =display-graphic-p= guard
would otherwise skip symbol fallbacks permanently, and EXWM creates four
workspace frames after the initial one.

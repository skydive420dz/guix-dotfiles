#+title: GuixPC P2.M1 Storage Maintenance Contract
#+date: 2026-07-16
#+status: design complete

* Outcome

P2.M1 separates SSD visibility, periodic maintenance, deep diagnostics, and
alerting into independently reviewable slices.  It does not treat maintenance
as a performance benchmark and does not combine it with generation pruning or
store garbage collection.

The design makes these decisions:

1. Validate and reuse the already-declared UDisks ATA SMART interface before
   adding =smartmontools= or another root daemon.
2. Make the first durable surface a repository-owned Scheme reader with an
   allowlisted, identifier-free human/JSON contract.
3. Keep UDisks D-Bus activation visible: a normal read may lazily start
   =udisksd= and perform read-only SMART metadata I/O; a cached-only mode must
   fail closed instead.
4. Treat periodic TRIM as an unselected policy.  Choose targets, a normally-on
   schedule, missed-event behavior, and a dry-run boundary before adding the
   Guix service.
5. Keep actual FITRIM, SMART self-tests, SMART enable/disable, a custom
   =smartd= daemon, alert delivery, package additions, generation cleanup, and
   =guix gc= outside this design slice.

P2.M1a now has an implemented, fixture-verified, and pushed source checkpoint.
Its separately authorized cached-only runtime gate passed; explicit completion
acceptance remains pending.  P2.M1b through P2.M1d remain unpromoted.

** Disclosed runtime-boundary deviation

The intended P2.M1 design review was source-only.  During delegated research,
an exploratory =udisksctl status= call used the well-known system D-Bus name.
That normally lazy-activated the already-declared root =udisksd= process at
=2026-07-16 16:23:34 EDT=.  Its initial housekeeping populated ATA SMART cache
at =16:23:35=.

This was a read-only device metadata operation, but it was a runtime change and
exceeded the announced design-only boundary.  It is retained rather than hidden:

- no System/Home declaration, package, generation, service definition, disk
  setting, SMART setting, self-test, mount, trim, reconfigure, generation
  activation, or reboot changed;
- the already-declared on-demand daemon became active and remained running;
- the first housekeeping read may wake a sleeping disk; this SSD was active;
- later housekeeping uses a no-wakeup path and treats a sleeping/busy device as
  a skipped refresh rather than a drive failure;
- raw =udisksctl status= output contained a drive serial and is rejected
  evidence.  No identity-bearing value is retained in this document.

The one-second daemon-start/cache-update sequence strongly supports normal
D-Bus lazy activation plus initial housekeeping, but no direct pre-query =ps=
row was captured.  It is therefore an inference, not a complete pre/post pair.

* Existing declarative SMART path

** Guix ownership

=guix/systems/guixpc.scm:78-100= inherits pinned =%desktop-services=.  Pinned
Guix includes =(service udisks-service-type)= at
=gnu/services/desktop.scm:2818-2821=.  The service definition at
=gnu/services/desktop.scm:1202-1238= extends:

- system D-Bus service activation;
- Polkit policy;
- udev rules;
- System activation state; and
- the System profile containing =udisksctl= and related artifacts.

The live System provides UDisks 2.10.1 and links =udisksd= against
=libatasmart= 0.19.  The D-Bus activation file runs the daemon as root.  This is
not a continuously started root Shepherd service; the first client of the
well-known D-Bus name can start it.

Pinned Guix also packages =smartmontools= 7.5, but neither GuixPC nor pinned
Guix currently supplies a =smartd-service-type=.  That package is not required
for the first visibility surface.

** Stable API surface

The installed =org.freedesktop.UDisks2.Drive.Ata= interface provides:

- =SmartSupported=, =SmartEnabled=, =SmartUpdated=, and =SmartFailing=;
- =SmartTemperature= and =SmartPowerOnSeconds=;
- current/past failing-attribute counts and bad-sector count;
- self-test status and percent remaining; and
- cached =SmartGetAttributes= tuples.

UDisks initial housekeeping performs a SMART metadata read when SMART is
supported and enabled.  The provider then schedules housekeeping every ten
minutes.  Later reads use the no-wakeup path; the cache can therefore become
stale when a disk sleeps or is busy.  Stale data must not be rewritten as a
health failure.

=SmartGetAttributes= returns already-cached attributes.  It does not request a
new device refresh.  =SmartUpdate=, =SmartSetEnabled=,
=SmartSelftestStart=/=Abort=, power-management calls, and security erase are
separate methods and are prohibited by P2.M1a.

* Privacy contract

** Rejected surfaces

Do not use or retain:

- =udisksctl status=, =info=, or =dump=;
- Drive =GetAll=, an object-manager dump, =busctl tree=, or broad introspection
  output from a serial-bearing drive path;
- a hard-coded Drive object path in source, captured argv logs, shell history,
  fixtures, documentation, or error messages;
- tracing such as =set -x= while resolving the Drive object; or
- raw model, serial, WWN, by-id path, filesystem UUID/PARTUUID, or host identity.

The fixed block object =/org/freedesktop/UDisks2/block_devices/sda= can be used
to resolve its Drive object privately.  The returned Drive path contains a
serial.  The selected =busctl= transport necessarily passes that object path as
one positional argument to each direct child invocation.  This is a transient
local-process-observation boundary: the path can be visible through process
inspection while the short-lived child exists.  The implementation must invoke
=busctl= directly without a shell and must never place the path in shell
history, tracing, durable argv capture, a fixture, a log, rendered output, raw
stderr, a diagnostic, or an exception.  P2.M1a protects retained and
user-visible evidence; it does not claim secrecy from root or same-user process
inspection during a call.  A future native Guile D-Bus transport could remove
this argv exposure, but is not required by the no-new-package slice.

** Allowlisted output

P2.M1a may expose only:

- the eleven named =Drive.Ata= properties in the retained sanitized summary table;
- selected cached attributes by stable attribute name;
- cache timestamp and calculated age;
- daemon activation policy (=allow= or =cached-only=); and
- schema/protocol version plus favorable/unknown/failing-indicator classification.

The implementation must fail on missing/extra property count, unexpected
D-Bus type, =SmartUpdated= zero, malformed JSON, an unknown selected attribute
unit, or any identifier-shaped forbidden key/value reaching the renderer.

* P2.M1a Scheme reader contract

** Command and implementation

Add a repository-owned Guile command, provisionally
=scripts/guix-disk-health=, that:

1. uses existing =busctl --system --json=short= as D-Bus transport;
2. parses JSON with the already-available Guile =(json)= module;
3. resolves the block-to-Drive relationship privately;
4. invokes =busctl= directly, without a shell, and requests individual
   allowlisted properties in fixed order;
5. calls cached =SmartGetAttributes= with an empty options dictionary;
6. converts Kelvin and seconds with exact labeled units;
7. prints a stable human view by default and an allowlisted JSON object on
   =--json=; and
8. never accepts a generic D-Bus method, property, object, or device argument.

The runner must discard or sanitize raw child stderr before constructing an
error because =busctl= may echo its object-path argument.  Only stable
identifier-free error codes and messages may reach either renderer.

The CLI should support:

- default =--allow-activate=: normal D-Bus activation is allowed and disclosed
  as read-only metadata I/O, not a zero-I/O query;
- =--cached-only=: pass =--auto-start=no= to =busctl= and fail with an
  =unavailable= state when UDisks is not already active;
- =--summary=: omit detailed attribute rendering while retaining the same
  cached-attribute collection and classification boundary; and
- =--json=: emit only the versioned allowlisted schema.

The implementation must not call =udisksctl= because its manual states that
the CLI is not a stable scripting interface.  It must not use =jq=, add a
package, require sudo, broaden =disk= group membership, add a setuid program, or
persist raw D-Bus output.

** Fixtures and gates

P2.M1a must include fixture-driven tests that never contact the live system bus:

- healthy selected-property and attribute fixture;
- =SmartUpdated= zero/unavailable fixture;
- stale-cache fixture that remains =unknown/stale= rather than =failed=;
- drive-failing and nonzero-counter fixtures;
- malformed count/type/JSON fixtures;
- ambiguous =success-or-never= self-test rendering;
- unknown-unit omission/failure behavior;
- forbidden identifier key/value fixture; and
- fake runner proof that only property =Get= and cached
  =SmartGetAttributes= are reachable.

The protocol cases use only an injected fake runner.  Separate lifecycle
probes spawn Guile itself to verify success, nonzero-exit output discard,
timeout before and after pipe EOF, output bounds, invalid UTF-8, signals, and
exec failure.  Neither path can resolve or execute =busctl=.

Validation remains split:

1. source/Guile syntax and fixture tests;
2. privacy/static command allowlist;
3. checkpoint and push;
4. one runtime =--cached-only= read while the daemon is already active;
5. a later ordinary-boot =--allow-activate= test that records pre/post daemon
   state without retaining raw output; and
6. no System/Home build, activation, sudo, device setting, self-test, or reboot
   merely to accept the script.

** Source checkpoint status

The P2.M1a source checkpoint adds the fixed launcher and Scheme module, one
synthetic response bundle, a 185-check focused gate, and strict-source gate
registration.  The production command performs only:

1. one fixed =Properties.Get= for the block object's Drive property;
2. eleven fixed =Properties.Get= calls for reviewed =Drive.Ata= properties;
3. one cached =SmartGetAttributes= call; and
4. complete protocol validation, classification, rendering, and a final
   identifier guard before its only output write.

Every child uses the fixed =busctl= executable and argument vector, a
five-second transport timeout plus a seven-second monotonic parent deadline,
bounded in-memory stdout, =/dev/null= stdin/stderr, and no shell.  A transient
private Drive path remains observable in child argv to a sufficiently
privileged concurrent process; the checkpoint does not claim to remove that
transport boundary.

Source, warning-clean compile, focused ShellCheck, exact-command,
classification, child-lifecycle, malformed-payload, privacy, and Org gates
pass.  No live reader invocation, system-bus query, UDisks activation, device
query, package, build, activation, privilege, trim, cleanup, or reboot is part
of this checkpoint.  The source/runtime evidence boundary is owned by
[[file:2026-07-16-p2.m1a-scheme-disk-health-source-checkpoint.org][the P2.M1a source-checkpoint record]].

* Periodic TRIM design

** Pinned Guix service semantics

Pinned =gnu/services/linux.scm:397-464= provides =fstrim-service-type=.  Its
defaults serialize exactly to:

#+begin_src text
fstrim --listed-in /etc/fstab:/proc/self/mountinfo \
       --verbose --quiet-unsupported
#+end_src

The default schedule is =0 0 * * 0=, Sunday midnight.  The root Shepherd timer:

- requires =user-processes=;
- runs as root because no user/group override is supplied to PID 1's Shepherd;
- waits for the child, preventing overlap;
- exposes a stateful =trigger= action;
- has no timeout, retry, persistent last-run state, durable service-specific
  log, or alert; and
- retains only bounded timer status/history in Shepherd memory/global logging.

Shepherd intentionally skips an overslept/missed calendar event and waits for
the next future match.  “Sunday-midnight trim opportunity” is accurate;
“guaranteed once every seven days” is not.  The schedule must use a time the
user normally leaves GuixPC awake.

Current =/etc/fstab= is nonempty, so =/proc/self/mountinfo= is only a fallback.
The listed candidates are =/=, =/home=, and the EFI mount.  The first design
should add =--types ext4= so unsupported EFI behavior is not silently treated
as accepted coverage.  =--quiet-unsupported= can suppress unsupported-ioctl
errors and clean their exit status; timer success alone cannot prove every
intended target was trimmed.

Recommended future declaration after schedule review:

#+begin_src scheme
(service fstrim-service-type
         (fstrim-configuration
          (schedule "USER-SELECTED-NORMALLY-ON-WEEKLY-TIME")
          (listed-in '("/etc/fstab" "/proc/self/mountinfo"))
          (verbose? #t)
          (quiet-unsupported? #t)
          (extra-arguments '("--types" "ext4"))))
#+end_src

The explicit fields pin intent against default drift.

** Dry-run and activation gates

P2.M1b may request one privileged dry-run:

#+begin_src sh
/run/setuid-programs/sudo \
  /run/current-system/profile/sbin/fstrim \
  --dry-run --verbose --types ext4 \
  --listed-in /etc/fstab:/proc/self/mountinfo
#+end_src

=--dry-run= skips the FITRIM ioctl.  It checks parsing and target selection but
does not prove permission, filesystem FITRIM support, actual discard, schedule
delivery, or future success.

P2.M1c may then add and build the explicit service.  Reconfigure normally loads
and starts the new timer without reboot; starting the timer waits for the next
matching occurrence and does not perform catch-up for a missed event.  A later
schedule/flag change does not restart an already-running changed service under
this pinned Guix revision; it needs an explicit privileged system-Shepherd
restart or reboot.

Bare =herd= over SSH targets Home Shepherd on this machine.  System timer
verification must use the privileged root Shepherd boundary.  A real
=herd trigger fstrim= invokes FITRIM and is a separately authorized hand test.
Rollback removes future scheduling but cannot undo completed discard requests.

** PARKED P2.M1d -- Durable SMART alerting or deep diagnostics

- First decide whether the UDisks reader plus later user-space alert consumer
  is sufficient.
- =smartmontools= may be considered for evidence-triggered deep diagnostics,
  using a one-shot no-serial capture rather than =smartd= by default.
- A custom =smartd= service is not selected: pinned Guix has no service type;
  =smartd= attempts to enable SMART at startup, logs identifiers, and requires
  separate state, privacy, notification, and privilege design.

* Explicit non-goals

P2.M1 design does not authorize:

- =SmartUpdate=, =SmartSetEnabled=, SMART self-test start/abort, offline data
  collection, autosave, standby/wakeup, secure erase, or generic D-Bus calls;
- =smartctl=, =smartd=, adding =smartmontools=, membership in =disk=, a setuid
  helper, or broad sudo policy;
- FITRIM, continuous mount discard, scheduler/read-ahead/write-cache changes,
  or treating discard as a performance benchmark;
- a System/Home build, reconfigure, activation, service restart, or reboot;
- generation deletion, reclaim estimation, store collection, automatic GC, or
  changes to the backup/retention gates; or
- retention of raw serial, WWN, model, object path, by-id path, UUID/PARTUUID,
  MAC address, or unfiltered command output.

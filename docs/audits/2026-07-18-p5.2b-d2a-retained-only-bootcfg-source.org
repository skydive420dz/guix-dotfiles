* Outcome

P5.2b-D2a implements and fixture-tests the configuration-aware, stage-only
GRUB construction path required by D-36.  It is a source checkpoint, not the
staged boot configuration and not a System-deletion checkpoint.

The implementation:

- evaluates GuixPC's real operating-system declaration only after binding the
  launcher, driver, module, directly loaded repository input, and complete
  Guix channel profile to reviewed bytes;
- extracts the complete bootloader object and validates the reviewed
  EFI/theme/gfxmode/terminal/keymap/menu contract;
- reads the accepted current System from generation 87's canonical store
  target, preserving a direct-store main entry;
- reads old boot parameters only through retained generation links
  =86,85,84,83,81,80,75=;
- provides a D2b stage action that invokes only the bootloader's
  configuration-file generator, Guix lowering, and immutable build
  realization;
- projects the accepted installed GRUB bytes down to those seven retained
  stanzas as an independent byte oracle; and
- exposes only =fixture-check= and an unprivileged =stage= action.

There is no installation, deletion, profile switch, GC-root promotion,
reconfiguration, activation, collection, or reboot action.

P5.2b-D2 is deliberately split into two publication checkpoints:

1. D2a is the publication checkpoint for this source, stage-only input
   specification, and fixtures.
2. D2b may run =stage= only after D2a is fetched clean, equal to
   =origin/main=, and still at the exact commit that first introduced the
   reviewed stage specification.  D2b will freeze the generated GRUB bytes and
   semantic result in the repository without making them active.

Live P5.2b remains =BACKLOG=.  Recovery roots, durable transaction journaling,
profile exclusion, link rollback, a fresh live deletion manifest, publication,
and manifest-specific attended authorization remain future gates.  P5.3
remains separate and =BACKLOG=.

* Pinned-Guix source decision

Pinned source review rejects =operating-system-bootcfg= for this task.  That
procedure synthesizes the main boot parameters from the source operating
system and can therefore make a prospective, unaccepted System output the
default entry.

The accepted path uses public APIs:

- =read-operating-system= and =profile-boot-parameters= from
  =(guix scripts system)=;
- =operating-system-bootloader=, =read-boot-parameters-file=, and
  =boot-parameters->menu-entry= from =(gnu system)=;
- =bootloader-configuration-file-generator= from =(gnu bootloader)=; and
- =lower-object= plus =built-derivations= for the uninstalled store build.

The current parameters are read from the canonical generation-87 System store
item.  Consequently, the main =gnu.system= and =gnu.load= arguments name that
direct store item.  Retained old parameters are read through their generation
links in normal newest-first order, so their entries continue to name the
reviewed rollback links.

The GRUB generator appends
=bootloader-configuration-menu-entries= itself.  D2a therefore passes only the
current main entry and fails closed unless the accepted-current and real-source
static menu-entry lists serialize equally.  The live probe proved both lists
are empty.

* Exact configuration contract

The real source object, not textual guesses from =grub.cfg=, must satisfy:

| Field | Required value |
|-------+----------------|
| Bootloader identity / name | exported =grub-efi-bootloader= / =grub-efi= |
| Target | =("/boot/efi")= |
| Default / timeout | =0= / =5= seconds |
| Terminal output / input | =(gfxterm)= / =()= |
| Serial unit / speed | disabled / disabled |
| Device-tree support / extra initrd | enabled / none |
| Static menu entries | zero |
| Theme image | present |
| Theme resolution | =(3440 . 1440)= |
| Theme gfxmodes | =3440x1440,2560x1440,1920x1080,auto= |
| Theme normal / highlight colors | =light-gray/black= / =yellow/black= |
| Keyboard | =us=; no variant/model; =caps:escape= |

This distinction matters.  The EFI target is consumed by a later bootloader
operation and is not encoded in =grub.cfg=.  Theme resolution and keyboard
options also influence derived assets rather than appearing literally in full.
The source tuple validates those fields; the byte oracle validates their
rendered consequences.

* Independent byte oracle

The byte oracle parses the SHA-bound accepted installed GRUB into:

1. the exact prefix through the old-configurations submenu opener;
2. the existing per-generation old-entry stanzas; and
3. the exact suffix beginning at the submenu close.

It preserves old stanzas =86,85,84,83,81,80,75= in that order and removes
every other old stanza.  A read-only live projection passed semantic

| Retained old-link occurrences | exactly 14; 2 per link |

* D2b stage gate

After this source checkpoint is published and fetched cleanly, D2b may run:

#+begin_src sh
cd ~/Projects/guix-dotfiles
git fetch origin
scripts/guix-system-retained-bootcfg stage \
  "$PWD/guix/machines/guixpc/p5.2b-d2-retained-bootcfg-stage.tsv"
#+end_src

The action must:

1. require clean =HEAD == refs/remotes/origin/main=, the reviewed parent as an
   ancestor, and =HEAD= equal to the exact commit that introduced the canonical
   D2a specification;
2. refuse root;
3. validate every bound source and live input before lowering;
4. build only an immutable =grub.cfg= store output;
5. require semantic success and byte equality with the projection;
6. revalidate System/Home/Pull links and pointers, current/booted System,
   source/channel/pin inputs, installed GRUB, and live =bootcfg= afterward;
7. write its result only to standard output; and
8. stop.

Realization can add immutable store objects and daemon build metadata.  D2b
must report that permitted store effect and must not claim whole-machine or
whole-store byte identity.  It creates no result GC root; the later tracked
artifact preserves the reviewed bytes if the unrooted store output is
collected.

dead/live GC query, collection, reconfiguration, activation, bootloader

* Handoff

Publish D2a first.  D2b then performs the unprivileged immutable build from the
clean published source, captures the exact generated bytes and result report,
and stops again.  Recovery transaction work follows as a separate successor;
live deletion remains blocked until that work, a fresh manifest, publication,
and manifest-specific authorization all exist.

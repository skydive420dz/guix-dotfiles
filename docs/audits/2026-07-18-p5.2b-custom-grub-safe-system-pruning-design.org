#+title: P5.2b Custom-GRUB-Safe System-Pruning Design
#+date: 2026-07-18
#+status: source and synthetic-fixture design complete; live System pruning held

* Outcome

P5.2b-D1 establishes a fail-closed design boundary for future Guix System
generation pruning.  It does not delete a live generation or expose a live
apply path.

The checkpoint:

- adds an exact-tuple System-link verifier and rollback primitive that can
  mutate only a sentinel-marked synthetic root;
- binds every fixture run to an explicit manifest SHA256;
- refuses root =/= and has no production =apply= action;
- proves staged deletion, rollback, partial-state recovery, injected failure,
  and signal-interruption behavior in an isolated fixture;
- changes =scripts/guix-generation-retention plan= so it no longer prints the
  unsafe supported System deletion command; and
- records the configuration-aware retained-only GRUB stage required before a
  live manifest can be produced or authorized.

P5.2, live P5.2b, Phase 5, and P5.3 remain =BACKLOG=.  This completed design
sub-boundary is not generation-deletion authorization.

* Exact truth boundary

This checkpoint implements and tests only an inert System-link manifest
verifier and rollback primitive against synthetic roots.  It neither
authorizes nor exposes live application.

It did not:

- run =guix system delete-generations=;
- unlink or retarget anything under the live =/var/guix=;
- rewrite =/boot/grub/grub.cfg= or the live =bootcfg= GC root;
- run =guix gc --list-dead=, =guix gc --list-live=, or a collection;
- remove a store object or reclaim storage;
- reconfigure, activate, or switch a System or Home profile;
- run a bootloader installer; or
- reboot.

One read-only =guix gc --references= query was used during design review
against the current =bootcfg= store item.  It queried ordinary forward
references only; it was not a dead/live query or a collection.  This is stated
explicitly because earlier work proved that =guix gc --list-dead= can prune
dangling automatic roots as a side effect.

* Confirmed pinned-Guix behavior

The pinned source review found the following order in
=guix/scripts/system.scm=:

1. =guix system delete-generations= deletes matching System generation links
   through =delete-matching-generations=; then
2. it unconditionally invokes =reinstall-bootloader=.

The relevant pinned-source ranges are:

- =guix/scripts/system.scm:400-440= for =reinstall-bootloader=;
- =guix/scripts/system.scm:1451-1458= for deletion followed by reinstall;
- =guix/scripts/package.scm:101-138= and =guix/ui.scm:2266-2270= for
  generation selection/deletion; and
- =guix/profiles.scm:2368-2390= for sequential link deletion.

This path is not transactional.  A failure after one link is removed does not
automatically restore earlier links, and the =guix system= deletion branch
does not acquire a profile lock.  =--no-bootloader= does not make this action
safe: the deletion branch ignores that option and still calls
=reinstall-bootloader=.

The reinstall path detects the bootloader type from the active System and
constructs a new default =bootloader-configuration=.  It does not consume the
full bootloader declaration in =guix/systems/guixpc.scm=.  GuixPC's declaration
owns settings the default record would lose:

- EFI target =/boot/efi=;
- the 3440x1440 theme resolution;
- the accepted gfxmode fallbacks; and
- the =caps:escape= keyboard layout.

The reinstall writes a new GRUB configuration and promotes a new =bootcfg= GC
root.  It passes =#:run-installer? #f=, so it does not need to rerun
=grub-install= or rewrite firmware variables; nevertheless, rewriting the live
GRUB configuration with default settings is outside the accepted boot
contract.

The apparently narrower command

#+begin_example
guix package -p /var/guix/profiles/system --delete-generations=...
#+end_example

is also rejected.  Although that path can reach link-only deletion, generic
package-profile processing continues afterward and expects a package
manifest.  A System profile has no such package manifest, so the command can
report failure after links have already been removed.

* Why exact-link-only deletion is not a completed design

The current GRUB file contains 85 unique noncurrent System generation-link
references, each appearing twice:

| Class | Generations | Link occurrences |
|-------+-------------+------------------|
| P5.2b candidates | 78 | 156 |
| retained old generations | 7 | 14 |
| total noncurrent link references | 85 | 170 |

The current generation 87 default entry uses its direct store path.  Deleting
the 78 candidate links while keeping GRUB byte-identical would therefore leave
78 advertised rollback entries broken immediately, represented by 156 stale
path occurrences.  The current default would remain bootable, but the menu
would no longer be truthful.

The read-only forward-reference query against the current =bootcfg= returned
only six direct registered references: the current System plus the
kernel/initrd and shared theme, keymap, and locale inputs.  It does not keep all
old System closures alive.  A later P5.3 collection could therefore collect
closures while stale GRUB entries still advertise them.

Consequently:

- leaving GRUB unchanged is not an acceptable postflight state;
- the current =bootcfg= cannot be treated as recovery coverage for every old
  System closure; and
- P5.3 remains blocked until the boot menu and retained System links agree.

** State machine

The verifier classifies the exact System-link inventory as:

| State | Meaning |
|-------+---------|
| =PREPARED= | all selected and retained tuples exist exactly |
| =APPLIED= | only the exact retained tuples exist |
| =PARTIAL= | any other inventory, including missing, extra, or retargeted links |
| =RECOVERY_REQUIRED= | a quarantine exists, regardless of the visible link inventory |

=fixture-apply= requires =PREPARED=.  It moves selected symlinks into a hidden
quarantine in the fixture profile, validates the =APPLIED= inventory, and then
unlinks the quarantined symlinks.  While staging or committing, exit and caught
signal traps restore every selected tuple.  If a saved symlink is already
gone, rollback reconstructs only its manifest-declared raw target after
proving the canonical fixture target still exists.

=fixture-rollback= refuses a clean =PREPARED= state, accepts =APPLIED=,
=PARTIAL=, or =RECOVERY_REQUIRED=, restores the exact selected tuples, and
refuses to overwrite an occupied path.  A populated quarantine with an
=APPLIED= inventory and an empty quarantine with a =PREPARED= inventory both
enter =RECOVERY_REQUIRED= and are reconciled rather than deadlocking apply and
rollback.

This state machine proves a primitive, not the future production transaction.
An uncatchable interruption leaves a detectable fixture quarantine that
=fixture-rollback= can reconcile.  A live transaction additionally requires
persistent recovery roots and a durable journal so recovery does not depend
only on ordinary store liveness or one directory.

The implementation uses =/bin/sh= control flow and is tested on the current
macOS development host and GuixPC.  It also depends on the host implementations
of =mktemp=, =realpath= or =readlink -f=, =sort=, =awk=, and SHA256 tooling; it
does not claim freestanding strict-POSIX utility portability.

* Rejected production designs

| Design | Rejection reason |
|--------+------------------|
| Supported =guix system delete-generations= | deletes first, then rebuilds GRUB from a default record that loses accepted GuixPC fields |
| Supported command plus =--no-bootloader= | deletion branch still invokes reinstall |
| Generic =guix package -p /var/guix/profiles/system= deletion | can fail in package-manifest processing after link deletion |
| Raw =unlink= loop | no lock, transaction, recovery roots, durable journal, or GRUB reconciliation |
| Exact-link primitive with byte-identical GRUB | leaves 78 broken advertised rollback entries / 156 stale occurrences |
| Generate bootcfg through a prospective source System | may make an unaccepted future System output the main boot entry |
| Delete links and defer GRUB cleanup to P5.3 | stale entries can outlive collected closures and misrepresent recovery |

* Required live design

A later P5.2b-D2 source checkpoint must build and validate the prospective GRUB
configuration before any link is touched:

1. load GuixPC's real operating-system declaration with the pinned Guix;
2. extract its full =operating-system-bootloader= configuration;
3. read boot parameters from the already accepted current generation 87, using
   that existing System as the main entry rather than creating a prospective
   System;
4. read old-entry boot parameters only for retained generations
   =75,80,81,83,84,85,86=;
5. lower and build a retained-only GRUB configuration without installing it;
6. prove that it preserves the EFI target, timeout/default policy, ultrawide
   theme/gfxmodes, keyboard layout, and accepted current direct-store entry;
7. prove that it contains exactly the seven retained old generation links and
   no candidate link; and
8. publish the exact staged bytes and semantic review result before creating a
   live deletion manifest.

Only a later manifest-specific attended stage may:

1. verify the pinned Guix revision, clean/upstream-equal repository, accepted
   current and booted System tuple, every generation-link tuple, pins, current
   GRUB hash, =bootcfg= target, and staged GRUB hash;
2. create persistent recovery GC roots for every candidate target and a durable
   phase journal;
3. revalidate immediately before mutation;
4. remove exactly the separately authorized System links with exact rollback;
5. install only the prebuilt configuration through the equivalent of
   =install-bootloader ... #:run-installer? #f=;
6. verify exact retained links =75,80,81,83,84,85,86,87=, current/booted
   generation 87, installed GRUB equality, the seven retained old entries,
   custom fields, unchanged EFI files/variables, and a promoted =bootcfg= root;
7. remove temporary recovery roots only after full postflight; and
8. stop without GC, activation, reconfiguration, profile switching, or reboot.

The production implementation must use a profile lock or an equally explicit
exclusion mechanism around the link transaction.  It must define recovery for
interruption at every journal phase, including before and after the GRUB-file
rename and =bootcfg= root promotion.

#+title: GuixPC P2.S1a Passive System, Build, and Storage Baseline
#+date: 2026-07-16
#+status: complete

* Outcome and boundary

P2.S1a establishes a passive, read-only baseline for GuixPC's system,
build, and storage state.  It observes the machine in its natural low-load
state and maps each important runtime value to repository, channel, kernel, or
platform ownership.

Rafael authorized this bounded slice with =proceed= on 2026-07-16.  The live
capture ran as the ordinary UID 1000 user.

The slice changed no declaration, package, mount, service, governor, scheduler,
swap setting, build policy, generation, garbage-collection state, or boot state.
No pre-existing service or graphical-session process was restarted, stopped, or
reconfigured; only transient SSH, shell, and collection processes ran.  It used
no privilege, synthetic load, benchmark, SMART self-test, trim command, cleanup,
activation, or reboot.

The evidence supports these conclusions:

- The point sample showed negligible CPU, memory, and I/O pressure, cool idle
  sensors, unused swap, about 35 GiB root/store free, about 139 GiB Home free,
  and low inode use.  This is not a whole-machine health claim or a load result.
- Periodic SSD TRIM has no current declarative or runtime owner.  This is a
  confirmed configuration with the maintenance policy still unselected, not
  evidence of current SSD failure or slowness.
- SMART health data was not captured.  =smartctl=/=smartd= are absent, but the
  inherited UDisks 2.10.1 service and =libatasmart= expose an ATA SMART D-Bus
  interface that P2.S1a did not activate or query.  Visibility is unverified,
  not proven absent, and no drive-health conclusion follows.
- Guix explicitly permits one local build job at a time and reports/allows up
  to four cores to that job; this is not hard CPU enforcement.  The fixed local
  cap's throughput and interactivity effects remain unmeasured.
- Current store and profile state does not establish cleanup urgency.  Forty-four
  unprotected older same-target generation-link candidates remain inventory
  evidence only.  Generation deletion and store collection are separate Phase
  5 operations behind the accepted backup and rollback gates.
- The current graphical process timestamps do not measure boot-to-desktop
  readiness because that session was deliberately restarted long after boot.
- No evidence from this point sample justifies changing the CPU governor,
  clocksource, transparent huge pages, swap or zram, I/O scheduler, read-ahead,
  Guix job count, or thermal policy.

P2.S1b measurement and P2.S1c experimentation remain =BACKLOG= and unselected.
No numeric performance target follows from P2.S1a.

This is deliberately narrower than the agenda's former broad P2.S1 heading:
passive observation is complete here; a controlled benchmark needs a separately
accepted workload and protocol; and any tuning or maintenance change needs its
own source, build, activation, runtime, rollback, and stop gates.  Generation
pruning and store collection are not performance tuning.

* Truth vocabulary

- =OBSERVED=: directly read from the synchronized live host during this slice.
- =CONFIRMED CONFIGURATION=: directly matched between source and runtime; it
  does not by itself establish performance impact.
- =POLICY UNSELECTED=: source and runtime confirm that no owner is configured,
  but the desired maintenance behavior has not yet been accepted.
- =HYPOTHESIS=: a plausible measurable effect that has not been tested.
- =UNKNOWN=: the evidence does not establish the value, history, or effect.
- =NOT AUTHORIZED=: no mutation, load, activation, cleanup, or reboot follows
  from this document.

A low-pressure point sample is not a claim that the machine is never
pressured.  An absent monitor is not a fault report.  A configured limit is not
a benchmark result.  An older same-target generation link is not a reclaimable
byte count.

** SMART ownership

Pinned Guix offers =smartmontools= 7.5, but that package is neither declared nor
installed and pinned Guix provides no =smartd-service-type=.  Those facts do
not establish that the current System lacks all SMART visibility.

The inherited =%desktop-services= includes =udisks-service-type=.  Runtime
evidence confirms UDisks 2.10.1 in the System profile, a root D-Bus-activatable
=udisksd=, linked =libatasmart=, and installed
=org.freedesktop.UDisks2.Drive.Ata= introspection.  The interface defines
=SmartSupported=, =SmartEnabled=, =SmartFailing=, =SmartUpdated=,
=SmartTemperature=, =SmartPowerOnSeconds=, failing/past-failed attribute counts,
bad-sector count, and self-test status properties.

P2.S1a intentionally did not activate =udisksd= or query a drive object.  It
therefore did not establish whether the installed SSD publishes that interface,
whether its cached data are current, which properties an unprivileged caller
may read, or whether the returned object contains device-unique identifiers.
Actual health, wear, error, and self-test data remain =UNKNOWN=.

The next read-only validation should query only an allowlisted ATA property set,
record whether D-Bus activation starts the already-declared daemon, and reject
serial, WWN, UUID, and raw-object output.  Only if that path is insufficient
should a later slice consider =smartmontools=.  Starting or aborting a SMART
self-test, enabling SMART, adding =smartd=, broadening block-device access, or
persisting raw identity-bearing output is not implied by “validate visibility.”

* Guix daemon and build ownership

** Confirmed runtime policy

The live daemon includes:

#+begin_src text
--cores=4
--max-jobs=1
--max-silent-time=3600
--timeout=86400
--log-compression=gzip
--discover=no
#+end_src

The configured substitute set is Bordeaux, Guix CI, and Nonguix.  No active
daemon builder child was present during the idle snapshot.

Pinned Guix documents =--cores=N= as allowing/reporting up to that many cores
to each build job and =--max-jobs=N= as the maximum number of parallel build
jobs.  The core count is exposed to builders and is not hard CPU enforcement.
The accepted interpretation is therefore:

#+begin_quote
Guix may run one build job at a time and reports/allows up to four cores to that
job when its build system honors the exported core count.
#+end_quote

This caps concurrent build jobs while reporting/allowing four cores to one
internally parallel build.  It does not constrain every daemon or substitution
activity, prove that normal operations are slow, prove that a different pair is
faster, or establish that four cores would be safe for each of several
simultaneous jobs.  Many normal operations use substitutes, which further
limits what can be inferred from configuration alone.

** Source ownership and drift risk

- =guix/systems/guixpc.scm:115-119= explicitly owns
  =--cores=4 --max-jobs=1=; the live daemon matches.
- =scripts/guix-build-gate= repeats the fixed concurrency in its candidate
  build paths.
- =scripts/guix-bootstrap:401-403= repeats it for bootstrap.
- Substitute URL constants also exist in the System/Nonguix assembly and in
  =guix-build-gate=, =guix-reconfigure=, and =guix-bootstrap=.  The effective
  set agrees today, while ordering and ownership differ.

The explicit cap and consistent gates are a strength: local build concurrency
cannot silently expand beyond the configured policy.  The duplicated literals
and missing recorded rationale are a configuration-hygiene risk.  Consolidation
is not justified inside a measurement slice because unchanged behavior would
need its own tests.

The Guix client revision in the current pull profile and the package hosting
the long-running daemon serve different lifecycle roles.  Their different
identities are not classified as a version mismatch without a demonstrated
protocol or behavior problem.

* Strong points

- Current and booted System converge, the live declaration matches repository
  source, channel pins are explicit, and the checkout stayed synchronized and
  clean through collection.
- Natural-state CPU, memory, and I/O pressure were negligible; idle
  temperatures were cool; swap was unused; and the snapshot recorded about 35
  GiB root/store and 139 GiB Home free with low inode use.
- CPU boost, ACPI idle states, discard capability, and ext4 error counters are
  visible through standard kernel interfaces without adding a monitoring stack.
- Guix build limits are explicit and consistent across the daemon and guarded
  build workflows, even though the policy has not been comparatively measured.
- Generation tooling inventories, protects, plans, and verifies without
  implementing a destructive apply path.
- Existing build and activation helpers preserve channel/source provenance and
  keep candidate build verification separate from live activation.
- The repository already has bounded log and invariant patterns that a later
  collector can reuse instead of inventing an unrelated framework.

* Weak points, unknowns, and non-findings

** Ownership and visibility limits

- Periodic TRIM has no current owner and remains a policy choice.  SMART already
  has an inherited UDisks path, but its usable property boundary is unverified.
- Build/substitute policy literals have multiple owners and can drift.
- Whole-system boot-to-ready, sustained thermal, and broad performance history
  are unselected observability limits, not proven defects.

** Unknowns that remain open

- actual SMART health, wear, error counts, and self-test history;
- last historical manual trim;
- sustained thermal behavior, fan response, and throttling;
- HPET's cost in real user workloads;
- scheduler/read-ahead comparative behavior;
- swap behavior under real pressure;
- Guix build throughput and interactivity under another concurrency policy;
- boot-to-visible-desktop-ready latency and its critical path;
- reclaimable store bytes after any future generation-link decision.

** Configuration changes not supported by this evidence

- replacing =schedutil= or =mq-deadline=;
- forcing TSC or another clocksource;
- enabling zram/zswap or changing swap/sysctl/THP values;
- changing read-ahead or write-cache policy;
- increasing =--max-jobs= or changing =--cores=;
- installing a thermal-control daemon solely because one exists;
- deleting generations, scheduling =guix gc=, or estimating reclaim from link
  count alone.

* Explicit non-goals and stop conditions

- a timed Guix build, =--no-substitutes= build, cache reset, download,
  substitute comparison, or change to cores, jobs, timeouts, or endpoints;
- a SMART short, long, or conveyance test or an assertion about drive health;
- =fstrim= or a discard, mount, scheduler, read-ahead, or write-cache change;

- a Home/System build or activation, service restart, EXWM restart, reboot, or
  cold-boot measurement;
- a persistent monitoring daemon, package addition, or alert policy.

Stop and split the work when observation requires privilege, sustained load,
cache manipulation, a device command with side effects, a new readiness marker,
an activation, or a reboot.  A missing metric stays =UNKNOWN= until that new
boundary is explicitly accepted.

* Successor choices and gates

No successor is promoted by this document.  The next decision may select one
of these narrow questions:

1. A storage-maintenance ownership slice: decide whether Guix should own a
   periodic trim schedule.  Package, service, schedule, privilege, build,
   activation, runtime, hand-test, and rollback decisions must be explicit.
2. A bounded UDisks SMART-visibility validation: query only allowlisted
   read-only ATA properties, capture D-Bus activation behavior, and retain no
   unique identifiers.  Validate the existing path before adding a package or
   daemon.
3. P2.S1b build measurement: choose one representative source-build workload
   and compare a small safe concurrency matrix without cache manipulation or
   policy activation.
4. P2.S1b boot instrumentation: define a user-relevant readiness boundary and
   retain timestamps across a separately approved ordinary cold boot.
5. P2.S1b thermal observation: only in response to a concrete thermal question,
   observe a bounded representative workload without an unreviewed stress test.
6. P2.S1c: after a measured problem exists, test at most one reversible change
   with an expected direction, stop condition, rollback, and unchanged-system
   acceptance gates.

Generation pruning and store collection are not choices in P2.S1b/P2.S1c.
They remain Phase 5 work behind restore-tested backup and an accepted retention
contract.

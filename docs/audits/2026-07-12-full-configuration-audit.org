* Audit boundary and integrity note

The audit covered the local checkout, the live checkout, active Guix System and
Home generations, the EXWM/Emacs session, user Shepherd services, editor tools,
shell syntax, application configuration, and Lisp tooling.

The intended audit was read-only.  No repository files, packages, channels,
generations, services, or machine configuration were changed.  Two incidental
state effects were discovered and reported during validation:

1. =scripts/emacs-check= loaded the complete init into the running EXWM Emacs.
   Named hooks and advice remained singletons, but the action was not
   observational and produced Org version-mismatch warnings.
2. A batch probe changed =HOME= but inherited the real =XDG_CACHE_HOME=.  Emacs
   reported pruning 20 unavailable recentf entries, which may have been saved on
   exit.  This proved that the current checker is not isolated.

The audit also exposed an elogind/SSH lifecycle leak.  Approximately 50 stale
sessions existed before the audit.  Short audit connections increased the count
to at least 91 closing sessions before new SSH connections were stopped.  Do not
run connection-heavy remote probes until Slice 7 establishes a safe procedure.

The following privileged areas were not available because local sudo
authentication is required:

- system Shepherd status through the protected socket;
- root-owned system, SSH, kernel, and Guix daemon logs;
- privileged confirmation of the elogind/PAM/sshd root cause.

Core processes were observed, but that is not evidence that every system service
is healthy.

* Strong points to preserve

- The README clearly separates System, Home, and application ownership.
- Guix owns packages; Emacs source does not install packages at runtime.
- Emacs modules are small, mostly focused, and use lexical binding.
- Elisp, Guile, and Common Lisp use appropriate native/REPL backends.
- Opening a Lisp file does not unexpectedly start a REPL.
- The shared =SPC l= surface gives the dialects a coherent user contract.
- Formatting is manual; there is no surprise format-on-save policy.
- Tracked fixtures and a tool-presence probe establish a useful base.
- Hardware-specific display/filesystem assumptions are centralized and
  documented for this one machine.
- Current repo-backed links were correct and source/deployed declarations
  matched.
- Display startup is accepted and stable at 3440x1440/100 Hz.
- Current resource headroom is excellent.
- No tracked secrets were found.

The live-symlink model is a deliberate strength for rapid configuration
iteration.  Preserve it unless rollback semantics are explicitly chosen over
immediacy; do not accidentally replace it with store snapshots during cleanup.

* Slice execution rules

Every slice must follow these rules:

1. Start from a clean working tree and record the current System, Home, and
   channel generations relevant to the slice.
2. Do not use repeated one-shot SSH connections until O-01 is closed.  Prefer
   local console work or one persistent connection when remote access is
   unavoidable.
3. Keep source edits, activation, and live acceptance as separate gates.
4. A passing evaluator/build is not live acceptance.
5. Never run a validation command that loads the complete init into the live
   EXWM session.
6. Isolate =HOME= and every XDG state/cache/config directory for batch tests.
7. Preserve current keyboard wiring and window behavior unless the slice
   explicitly owns a change.
8. Do not expand dialect breadth while a P0/P1 prerequisite for that layer is
   open.
9. Update this document when a slice is accepted, rejected, or split.
10. Record exact keep/delete and rollback criteria for generation, package, and
    state cleanup.

When a slice is accepted, change its heading to =DONE=, add a =completed=
property, and append an acceptance record using this shape:

*** Investigation and recovery record: 2026-07-12

The source/build gates, controlled activation, reboot persistence, local
desktop checks, and bounded SSH negative controls are complete.

Pre-change evidence was captured through one persistent SSH connection only:

- at =2026-07-12T14:40:56-04:00=, elogind held 96 sessions: 94
  leaderless sessions in =closing= plus the active graphical and control
  sessions;
- representative stale session =c502= had =Service=sshd=, a dead leader, no
  =Scope=, and no remaining reference FIFO;
- the control session was =c514=; several commands inside it left the session
  count unchanged at 96;
- =/run/booted-system= was the preserved generation 75 closure
  =8zvbahq7s52fm1f27rc3qhbp02kj0zxh-system=, while
  =/run/current-system= was generation 80
  =jk1793wbqbn68y2s8ccpnpqicibrf8nz-system=;
- Home generation 30 was current, while D-Bus, PipeWire, WirePlumber, and
  PipeWire-Pulse all reported replacement pending;
- System generations 75, 79, and 80 and Home generations 29 and 30 were
  confirmed present.  No generation cleanup was performed.

The generated SSH service uses the stock Guix OpenSSH configuration and a PAM
session stack containing =pam_elogind.so=, =pam_loginuid.so=, =pam_env.so=,
and =pam_unix.so=.  The running elogind was vanilla 255.17 on cgroup v2.  The
project's existing elogind overrides only govern idle, lid, and suspend policy;
none owns session garbage collection.

The observed failure strongly matches two upstream 255.17 reports: short SSH
commands leave dead-leader sessions indefinitely in =closing=, and sessions can
remain closing without any process.  Upstream 255.24 explicitly fixes garbage
collection after FIFO EOF, leader exit, and release timeout:

- [[https://github.com/elogind/elogind/issues/297][elogind issue 297: SSH sessions remain closing]];
- [[https://github.com/elogind/elogind/issues/327][elogind issue 327: closing session without a process]];
- [[https://github.com/elogind/elogind/releases/tag/v255.24][elogind 255.24 release notes]].

The exact upstream match plus the post-upgrade cleanup result establishes the
255.17 session-GC defect as the accepted root cause for this host.  Do not use
=loginctl terminate-session= or =kill-session= on 255.17, restart elogind from
the only remote connection, delete files below =/run/systemd/sessions=, or mask
this class of failure with =KillUserProcesses= or disabled PAM.

Commit =54d8408a30e650291ae40637b02c781a19bfb0ee= introduces a narrow
service-owned elogind 255.24 override.  It uses upstream commit
=8b6e5172370b9864b3e86907e5b9a9cc6153b7e3= and recursive hash
=0id4gjwjxcb860ibpr5irc3birgr89m650q4alrkpqdnd81nfyvk=.  The package and
complete system build passed.  The candidate system is
=kk8k5a84wzypz6dhpgi8m0gf6mkiid9x-system=; its profile =loginctl=, Shepherd
daemon, =pam_elogind.so=, D-Bus metadata, udev data, and polkit data all resolve
to =ff2932sq8k8z5vir6pav7zlkj5615gcp-elogind-255.24=.  The generated candidate
=sshd= PAM service also resolves to that exact module.

Activation must be performed from a local console because remote noninteractive
=sudo= is unavailable.  Between reconfigure and reboot, PAM would point at the
new module while the old daemon could remain replacement pending; make no SSH
or other fresh PAM login in that interval.  From tty2 or another verified local
console, run:

#+begin_src sh
sudo /run/current-system/profile/bin/sh -c \
  '/home/skydive420dz/.config/guix/current/bin/guix system reconfigure \
     /home/skydive420dz/Projects/guix-dotfiles/guix/systems/guixpc.scm &&
   /run/current-system/profile/sbin/reboot'
#+end_src

The single privileged shell is intentional: its PAM session starts before
activation, and it performs the reboot without opening a second PAM session in
the temporary mixed-daemon interval.

Keep every current generation until acceptance.  If the candidate fails, boot
generation 80 from GRUB or set it from a recovery console and reboot:

#+begin_src sh
sudo /run/current-system/profile/bin/sh -c \
  '/home/skydive420dz/.config/guix/current/bin/guix system switch-generation 80 &&
   /run/current-system/profile/sbin/reboot'
#+end_src

Generation 75 remains the last observed working boot target.  Home generation
29 remains the Home fallback.  Do not use =roll-back= when a specific target is
required: from the candidate it would select the immediately preceding
generation, not necessarily 80 or 75.

Post-reboot acceptance first confirmed the local desktop, converged
generations, running elogind 255.24, and fresh Shepherd services.  One SSH
control connection then held a stable session count while repeated commands
ran inside it.  Separate command and PTY sessions were both absent at the first
observation and after 30 seconds; the total returned to baseline with zero
=closing= sessions.

#+title: GuixPC P3.1 Visual Ownership Baseline and Semantic-Token Contract
#+date: 2026-07-16
#+status: accepted complete

* Outcome

P3.1 establishes the current visual ownership map and a bounded architecture
for making it coherent.  It changes no palette, font, theme, cursor, icon,
wallpaper, compositor, package, Home/System declaration, generation, or live
process.

The audit finds a useful partial baseline rather than a broken desktop:

- Emacs, Kitty, Fish, Xinit, Picom, and wallpaper have identifiable repository
  owners and live entrypoints.
- The live Emacs default background matched Kitty's declared canvas literal,
  but no running Kitty parser was sampled; the rest of their palettes and
  their different transparency mechanisms are not under one contract.
- GTK 3/4, icons, and cursors have no declared GuixPC visual policy.  Their
  current behavior is toolkit/profile fallback behavior.  JetBrainsMono Nerd
  Font is accepted for a later migration but is not yet active.
- Qt has no selected application or current configuration and remains outside
  the target set.
- The later ownership migration must preserve the accepted color/transparency
  result, deliberately migrate font roles to JetBrainsMono Nerd Font, and keep
  the current wallpaper.  Every still-open visual choice requires explicit
  acceptance, and all visual changes require real-display acceptance.

The proposed implementation uses a data-only Scheme token source, a pure
validator/renderer module, deterministic complete target files, and a later
atomic ownership migration through Guix Home.  In particular, Kitty must not
use an include: the repository's security policy rejects every Kitty include
form.  A later renderer must produce the complete immutable =kitty.conf=.

P3.1 is documentation and design only.  Checkpoint
=893382750e8d8896ffde9e947c9e699f18c999ef= was published with clean
upstream equality, and Rafael explicitly accepted the contract after reviewing
its overview.  No visual implementation or runtime action was required for
closure.

The first post-contract choices are also explicit: use JetBrainsMono Nerd Font
globally and keep the current wallpaper.  These are P3.2/P3.3 visual-migration
inputs rather than retroactive P3.1 implementation.  The remaining production
choices were left open at P3.1 closure for a reviewed P3.2 decision gate; the
post-contract addendum below records their later acceptance.

* Scope and truth boundary

** In scope

- verify the actual GuixPC host, synchronized checkout, active System/Home,
  and live visual entrypoints;
- map declared and live ownership for Emacs, Kitty, Fish, GTK 3/4, fonts,
  icons, cursor, X session, compositor, opacity, and wallpaper;
- distinguish an observed runtime value from a source declaration, packaged
  default, fallback, inference, or user choice still required;
- define a semantic schema, renderer boundary, validation rules, migration
  sequence, rollback, and stop conditions; and
- promote only the documentation/design slice.

** Explicitly out of scope

- editing any visual source value or live configuration;
- adding a package, theme, font, icon set, cursor, wallpaper, daemon, or CSS;
- rendering or wiring generated files;
- running a Guix build, =guix home reconfigure=, =guix system reconfigure=,
  generation activation, EXWM reload/restart, GUI launch, or reboot;
- changing or restarting Emacs, Kitty, Fish, Picom, Xwallpaper, GTK, or X11;
- taking a screenshot or claiming current root-window pixels were sampled;
- selecting production visual values inside P3.1; accepted post-contract
  choices are recorded only as closure evidence and later-slice inputs; and
- producing a Qt adapter, Qt environment variable, or Qt package.

** Evidence vocabulary

- =DECLARED= is found in the synchronized repository.
- =GENERATED= is emitted by the active Guix Home/System artifact.
- =OBSERVED= is read from the live host or graphical Emacs process.
- =PACKAGED DEFAULT= is read from the installed package, not owned by this
  repository and subject to change on package upgrade.
- =INFERRED= follows from bounded evidence but was not directly sampled.
- =UNMEASURED= or =UNOWNED= must not be rewritten as a defect or a chosen
  policy.

The installed =kitty-latest= is 0.47.4.  No Kitty process was open during the

=UNMEASURED=.

Only Kitty's declared canvas numerically matches the observed live Emacs

The two transparency controls are intentionally different quantities:

- Kitty uses a background-opacity ratio of =0.0=; and
- Picom applies whole-window opacity =85= percent to Emacs.

The user accepts the combined visual result.  A semantic contract must preserve

* Semantic-token contract

** Ownership model

The canonical input should be =theme/tokens.scm=: data only, primitive-loadable
without importing Guix modules, starting processes, reading the network, or
consulting mutable user state.  A plain quoted alist or equivalently inert
Scheme data is preferred.

=guix/modules/sk/theme.scm= should provide pure validation and rendering.  It
must accept explicit input data and return deterministic strings/files; module
import must not inspect the live machine or mutate state.

The source distinguishes four kinds of data:

1. semantic roles shared where their meaning actually matches;
2. typography and desktop-policy identities;
3. target calibrations with explicit units; and
4. repo-relative visual assets.

It must not encode target syntax in the canonical data or pretend that every
toolkit implements the same property.

** Required semantic roles

The initial exact schema should contain:

- =canvas=, =surface=, and =surface-raised=;
- =text=, =text-muted=, and =text-disabled=;
- =accent= and =on-accent=;
- =selection= and =on-selection=;
- =focus=;
- =success=, =warning=, and =error=;
- =border= and =shadow=; and
- =cursor= and =on-cursor=.

These are UI roles, not a complete syntax-highlighting palette.  Emacs syntax
faces and terminal ANSI colors need explicit adapter policy: either map them
from a reviewed extended role set or preserve the current theme/default
identity during the first migration.  The renderer must never silently invent
unlisted colors.

** Typography and desktop policy

The schema should name:

- fixed-width family;
- proportional/UI family;
- UI size with an explicit unit;
- ordered symbol/emoji fallback families;
- GTK base theme identity;
- desktop icon-theme identity;
- cursor-theme identity and cursor size.

The first implementation may not fabricate unresolved values.  It should
either require accepted values before emitting a target or deliberately retain
the current target behavior with that boundary represented and tested.

** Target calibrations

Target-specific quantities must preserve units in their key names or validated
records:

- Emacs default face height in tenths of a point;
- Kitty font size in points;
- Emacs whole-frame/Picom opacity in percent; and
- Kitty background opacity as a ratio.

The captured current values are live Emacs height 120, Kitty's declared 14
points, the active Picom rule's 85 percent, and Kitty's declared 0.0 ratio.
They are accepted P3.1 preservation inputs; exact rendered ownership remains a
P3.2/P3.3 gate.  Sharing an untyped =opacity= key is prohibited.

** Assets

The schema may contain a repository-relative wallpaper path and an enumerated
fit mode.  Absolute Home paths, store paths, URLs, mutable external state, and
command strings are not token values.  Asset existence and safe path
containment must be validated from an explicit repository root.

** Target set

The exact initial render-target set is:

1. Emacs adapter data/source;
2. complete Kitty configuration;
3. Fish color/prompt fragment;
4. GTK 3 settings;
5. GTK 4 settings; and
6. X-session/compositor/wallpaper data.

The validator must reject unknown target names and Qt output.  Chromium is a
GTK smoke-test consumer, not a separate website-theme renderer.

* Rendering and wiring contract

** P3.2 offline outputs

P3.2 may add the pure schema/validator, renderers, and synthetic complete
fixtures without wiring or activation.  Those fixtures must be unmistakably
non-production and complete enough to prove:

- deterministic output across two renders;
- correct escaping and syntax per target;
- no timestamps, absolute Home paths, store paths, host identity, or mutable
  state;
- exact role/calibration propagation; and
- no Qt artifact.

At P3.1 closure, a production =theme/tokens.scm= and complete production
outputs were blocked on unresolved policy choices.  If those decisions were
accepted before P3.2 began, production data could join the same checkpoint;
otherwise P3.2 would remain fixture-only and P3.3 could not start.  GTK had no
declared policy that a renderer could honestly preserve as a production no-op.
Those decisions are now accepted.  P3.2 first published a fixture-only
checkpoint and is adding the production datum in a second source-only
checkpoint; this does not retroactively add implementation to P3.1.

For Kitty, combine a reviewed static nonvisual/security base with token-derived
values to render one complete file.  Do not weaken the existing
=desktop-policy-check= or introduce any include form.  Preserve trail timing
as reviewed Kitty-specific policy while mapping its visual trail color from an
explicit semantic role rather than retaining the named =blue= literal.

For Emacs, preserve early theme/font ownership so Org or another package cannot
load a conflicting built-in/newer theme path before the adapter.  Loading the
adapter must be testable in isolated batch Emacs without starting EXWM.

For GTK, prefer stable settings and a named dark base over broad widget CSS.
GTK 3 and GTK 4 outputs may differ syntactically while consuming the same
approved roles.  The design does not promise pixel identity.

** P3.3 immutable Home wiring

P3.3 should wire outputs through Guix Home and build the candidate before any
live activation.  Kitty requires an atomic ownership migration:

1. render a complete candidate config;
2. pass the Kitty security and syntax/static gates;
3. remove the current repo-link ownership in the same source change that adds
   immutable Home ownership; and
4. prove exactly one generated live target exists after activation.

Source validation, Home evaluation/build, checkpoint/push, activation, and
runtime inspection remain separate gates.  Building a Home candidate does not
mean it is live.

** P3.4 live visual acceptance

Only after the source checkpoint is pushed should a separate activation and
live slice inspect:

- Emacs/EXWM normal, inactive, selected, prompt, warning, and error states;
- a real Kitty process, ANSI/selection/cursor/tab states, and remote-control
  security invariants;
- Fish prompt, syntax, autosuggestion, and pager states;
- one GTK 3 application, one explicitly selected GTK 4 application, and
  Chromium UI without forcing website colors;
- cursor and icon fallbacks;
- Picom/wallpaper ownership, real-screen contrast, and the ultrawide result;
  and
- adjacent SSH, desktop-session, and EXWM stability gates.

A cold reboot is required only if startup/session persistence is part of the
accepted wiring.  Visual acceptance requires Rafael at the real display; a
build or SSH query cannot substitute for it.

* Validation contract

The offline checker should fail on:

- missing, unknown, or recursively duplicated keys;
- any color outside canonical lowercase =#rrggbb=;
- incorrect types, invalid enum values, unsafe relative paths, missing assets,
  or out-of-range sizes/opacities;
- unrecognized font/theme names after a deterministic candidate-profile lookup
  is available;
- timestamps, nondeterministic ordering, absolute Home paths, or live-state
  reads in rendered output;
- a target set other than the exact six-surface set;
- any Qt output, environment variable, package, or adapter;
- any Kitty include or unsafe remote-control surface;
- raw in-scope colors, font families, opacity values, or wallpaper paths left
  outside approved source/adapter boundaries after migration; and
- adapter output that changes when the same source is rendered twice.

Raw-literal scanning must explicitly exclude documentation, fixtures/test
expectations, generated golden files as appropriate, and reviewed artwork such
as SVG logos.  Otherwise the checker would confuse evidence and asset pixels
with configuration drift.

The checker should also run target-specific parsing/load gates:

- Fish syntax;
- isolated Emacs load/face assertions;
- Kitty config/security assertions without requiring a GUI process;
- GTK INI/settings syntax for both versions; and
- X-session shell/static ownership checks.

Contrast validation should cover agreed foreground/background pairs only after
Rafael selects a floor.  Transparent target contrast also requires the later
wallpaper/display hand test.  P3.1 records this unresolved decision rather than
silently choosing WCAG numbers for desktop chrome.

* Migration invariants, rollback, and stop conditions

** Accepted migration invariants

- Preserve the observed live Emacs / declared Kitty =#0d0e1c= baseline and
  D-06's accepted visual-result intent through the first ownership migration;
  the exact palette-role mapping is frozen by the later P3.2 production
  decision.
- D-05 is now complete: migrate every declared font role to the appropriate

** Rollback

P3.1 rollback is documentation-only: revert its checkpoint.  It has no runtime
generation to roll back.

P3.2 rollback removes unused offline source, tokens, and fixtures; it must not
own live files.  P3.3/P3.4 must record the exact previous Home generation and
source checkpoint before activation.  Because current Kitty ownership is a
direct repository link, rollback must restore both source ownership and the
prior Home generation when relevant; switching only one side is incomplete.

** Stop conditions

Stop before implementation or activation if:

- the renderer requires a Kitty include or weakens its security checks;
- more than one owner would write the same live target;
- unresolved palette/GTK/icon/cursor/scaling/contrast/Fish/test-surface choices
  are guessed;
- a build is represented as live activation;
- GTK support requires broad unreviewed CSS or a Qt dependency;
- deterministic rendering or raw-literal drift checks cannot distinguish
  source from assets/evidence;
- visual comparison cannot be performed at the real display; or
- the first migration changes the accepted appearance without an explicit
  visual-change decision.

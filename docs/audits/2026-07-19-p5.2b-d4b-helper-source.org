* Constructor boundary

[[file:../../guix/modules/sk/system-pruning-fused-artifact.scm][=(sk system-pruning-fused-artifact)=]]
has only one artifact-construction entry point: the repository-taking
=sk:published-d4a-fused-artifact=.  It revalidates the frozen renderer and
every input, then independently asserts the final =95b84e2...= / 956,987-byte
render before constructing one =computed-file=.  The generic input-taking
constructor is private, so the reviewed build expression cannot be redirected
to a caller-supplied alternate manifest.

The builder writes the exact rendered string once with UTF-8 encoding and
changes only the output mode to =0555=.  It does not interpolate its final
store path into its own contents, so no self-reference cycle exists.
=#:local-build? #t=, =#:graft? #f=, and =#:substitutable? #f= are explicit.

[[file:../../guix/machines/guixpc/p5.2b-d4b-fused-program.scm][The build-only expression]]
resolves the repository from its own filename and returns only the closed
file-like object.  It also refuses an accidental direct =guix build -f=
without the reviewed acceptance-token schema.  Merely evaluating the separate
pure checker constructs an inert record; it does not call =lower-object=,
=gexp->derivation=, a store connection, or a build operation.

* Preflight and realization-gate design

[[file:../../tests/guix-system-pruning-d4b-preflight][The read-only preflight]]
requires the exact GuixPC host, UID, repository path, clean
=HEAD == origin/main=, and published D4a ancestry.  It combines:

- the exact 3,244-byte D4a selected-live helper and its 19-row result; and
- a sorted filesystem inventory of every top-level =/gnu/store= name.

It hashes one canonical packet containing those two identities, the helper
commit, rendered-program identity, host, UID, and system.  It performs no
=guix gc= or daemon query and reports =authorization=NOT-GRANTED=.  The
resulting one-line acceptance token additionally fixes exactly one lowering,
one realization-or-confirmation, no substitutes, grafts, offload, result/GC
root, or live action.

[[file:../../tests/guix-system-pruning-d4b-realize][The future realization gate]]
accepts only that complete token and no pass-through Guix options.  It
recomputes the packet immediately and refuses stale live/store state before
lowering.  Before that preflight it exports the exact accepted =HEAD= with
sanitized Git into a private =/tmp= tree, makes the exported tree read-only,
and records its Git-archive identity plus reproducible metadata/file-digest
manifests.  The constructor, module load path, derivation-output reader,
derivation-plan reader, and failure-time selected-live helper are all
consumed from that frozen export, rather than from the mutable working tree.

The gate invokes the exact pinned Guix command beneath =env -i= with isolated
=HOME=, =TMPDIR=, and =XDG_CACHE_HOME=; this excludes inherited
=GUIX_BUILD_OPTIONS= as well as Guile paths.  Its two fixed phases are:

1. =guix build --derivations -f= of the exact expression; then
2. =guix build= of that exact derivation path.

The second phase does not reevaluate or relower the expression.  Neither
phase contains =--root=.  One separate warning-fatal Scheme reader accepts
only one derivation output named =out=.  A second reader recursively
enumerates the exact lowered derivation, its outputs, sources, builders, and
input derivations.  After a future accepted operation, the gate will retain
raw pre/post live snapshots, store-name inventories, every fixed Guix and
reader stdout/stderr channel, the final result packet, and exact
additions/removals under one bounded =/tmp= evidence directory.  Every
top-level store addition must belong to that exact derivation plan; any
unrelated addition fails the gate.  The plan is a conservative
reachable-derivation-graph bound: a matching path is permitted by the exact
plan, not claimed to have been causally produced by this one invocation.

After the build, a normal commit-bound postflight and a second independent
snapshot from the frozen selected-live helper plus direct =/gnu/store=
inventory must agree.  The frozen source manifests and the clean
=HEAD == origin/main= proof are repeated immediately before PASS.

On failure after lowering starts, the exit trap reuses the already
independently agreed normal postflight when available.  Otherwise it bypasses
compound Git preconditions and directly captures selected-live and store
state.  It persists each phase status, computes additions/removals, and labels
every addition =DERIVATION-PLAN=, =UNRELATED=, or =UNCLASSIFIED= when a valid
plan was not yet available.  This also runs after an otherwise successful
postflight if a later invariant fails.  No part of this gate has run in this
source checkpoint.

* Exact-code loader proof design

[[file:../../tests/guix-system-pruning-fused-loader-behavior-check.scm][The loader behavior checker]]
does not alter the artifact and does not require a second store output.  In a
fresh Guile process it:

1. reads forms directly from the accepted canonical store program;
2. preserves that real filename on the input port, so the frozen
   =current-filename= location and ownership guard executes normally;
3. evaluates only the exact startup/loader prefix;
4. parses and verifies, but never evaluates, the frozen five-form production
   tail; and
5. calls those exact preflight/evaluator/dispatcher procedures with closed
   canary packets or the frozen production packets.

Once executed against the accepted store program, its four isolated canary
modes must prove:

- =malformed-final=: a digest-valid final packet declaring the wrong module
  fails before any initializer or main call;
- =preexisting-final=: a preexisting expected driver module retains its exact
  object and marker while every other expected module remains absent;
- =wrong-digest=: a canonical packet carrying the wrong SHA256 fails at the
  exact source-digest check before any initializer, main call, or project
  module exists; and
- =success=: all ten modules initialize in canonical order, each evaluated
  module equals its registry object, the caller module is restored, and
  =sk:main= is invoked exactly once.

Its separate =production-suite= mode must load the actual ten production packets
from the verified tail, initialize their exact modules, and run one externally
frozen D4a suite in the same fresh process.  The runtime gate will invoke that
mode separately for boundary, root backend, orchestrator, reconciliation,
phase engine, and fixture runtime.  Each suite's SHA256/size is frozen and
exactly one suite PASS plus one loader PASS is required.  The seventh D4a
renderer suite remains the source gate because the renderer is deliberately
not a production module embedded in the artifact.

This is an exact-code behavioral proof around the one accepted artifact, not
a claim that a separately generated probe is production-equivalent.

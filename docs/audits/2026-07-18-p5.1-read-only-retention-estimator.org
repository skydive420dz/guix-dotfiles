* Outcome

P5.1 adds a new =estimate= action to
=scripts/guix-generation-retention=.  It implements the D-34 planning floor
without changing the older =plan= action:

- =plan= remains the narrow same-target duplicate-link reporter;
- =estimate= retains the current closure, the booted System closure when it
  differs, up to five newest additional distinct closures, and every explicit
  pin; and
- =estimate= classifies every other System, Home, and Pull generation link,
  simulates removal of those roots, and measures forward-reference NAR bytes
  through a Scheme helper.

The exact live result is preserved as
[[file:data/2026-07-18-p5.1-retention-estimate.tsv][the 329-row P5.1 estimator snapshot]].  Two consecutive runs on unchanged
state were byte-identical.  The tool and its live acceptance invoked no Guix
collector mode and changed no generation link, store object, profile pointer,
pin, configuration, activation, disk, backup, boot state, or persistent
process/session state.

The byte results are deliberately not called reclaim predictions or upper
bounds.  They cover ordinary forward references and NAR serialization sizes.
Guix derivation-retention policy, process roots, temporary roots, hardlink/block
allocation, and collector-time races are not modeled.  The only guaranteed
reclaim figure is therefore exactly zero.

=93619e90bfb01a43a1c4858be9a09ef76f818eb6=.  The actual checkout is clean,
=HEAD= equals =origin/main=, and ahead/behind is =0/0=.  P5.2 generation-link
deletion and P5.3 store collection remain unpromoted and require separate

* Retention policy

For each System, Home, and Pull profile, closure recency is the greatest
generation number targeting that canonical store closure.  The retained set is
the union of:

1. the current closure and current generation link;
2. the booted System closure when it differs from current;
3. up to five newest additional distinct closures, excluding current and
   booted before selecting those slots; and
4. every exact generation and closure in =generation-pins.tsv=.

Pins do not consume or replace the five recency slots.  A retained closure
keeps at least one generation link.  If the booted closure has multiple
generation links, every matching link is retained and reported as ambiguous
rather than guessing which generation number supplied the boot.

Every non-retained link is classified as:

- =duplicate-retained-closure= when another retained link keeps the same
  closure; or
- =unretained-closure= when the closure is outside the policy floor.

The Scheme helper simulates removal of all candidate generation roots,
including duplicate links, and fails unless every retained closure remains
directly rooted afterward.

* Implementation and API boundary

=scripts/guix-generation-retention= owns inventory, pin and pointer
validation, deterministic policy selection, machine-readable output, and
before/after fingerprints.  The existing =inventory=, =plan=, =verify=, and
=rollback-drill= behavior remains available.

=scripts/guix-generation-retention-estimator.scm= runs through the currently
selected Guix and imports only these store operations:

#+begin_example
find-roots
valid-path?
requisites
query-path-info
path-info-nar-size
with-store
#+end_example

The helper validates absolute regular input files, canonical local store
items, ignored-root membership, retained-root survival, local path validity,
root snapshot stability, and non-negative NAR sizes.  It buffers output until
the second root snapshot matches the first.

The fixture statically rejects collector, root-registration, substitution,
build, and realization operations, including =run-gc=, =dead-paths=,
=live-paths=, =delete-paths=, =add-temp-root=, =add-indirect-root=,
=build-things=, and =ensure-path=.  Neither the shell action nor the helper
contains an alternate mutating path.

* Output truth table

| Record | Meaning | Reclaim interpretation |
|--------+---------+------------------------|
| =profile= | exact link count, distinct-closure count, current generation and target | inventory only |
| =closure= | retained/candidate closure with newest generation and reasons | policy result |
| =retain= | protected generation link and reasons | must survive P5.2 |
| =candidate ... duplicate-retained-closure= | removable link whose target closure remains retained | zero store release from that link alone |
| =candidate ... unretained-closure= | link outside the D-34 floor | exact P5.2 review candidate, not authorization |
| =known-root= | daemon-known persistent root after hypothetical candidate-link removal | incomplete collector-liveness view |
| =candidate-closure-union= | union of ordinary forward requisites for candidate closures | measured NAR footprint, not reclaim |
| =candidate-exclusive-vs-policy= | candidate forward requisites absent from retained-policy forward requisites | forward-reference difference only |
| =candidate-exclusive-vs-persistent-known-roots= | candidate forward requisites absent from the simulated persistent-root forward closure | forward-reference difference only |
| =guaranteed-reclaimable= | conservative guaranteed lower bound | always =0 paths / 0 bytes= under current coverage |
| =coverage= | modeled and unmodeled surfaces | mandatory qualification |

The shell validator requires one of every estimate and coverage record, exact
semantics labels, a hard =0/0= guaranteed lower bound, and monotonic path/byte
counts.  Malformed or incomplete helper output fails closed.

* NAR accounting and coverage limits

After hypothetically ignoring all 108 candidate generation roots, 72
daemon-known persistent roots remained.  The helper observed:

| Metric | Paths | Exact NAR bytes | Approximate GiB | Semantics |
|--------+-------+-----------------+-----------------+-----------|
| Candidate closure union | 3,845 | 27,363,164,296 | 25.48 | measured forward-reference footprint |
| Candidate-exclusive vs retained policy | 2,535 | 11,299,406,400 | 10.52 | forward-reference difference only |
| Candidate-exclusive vs persistent known roots | 2,534 | 11,299,357,328 | 10.52 | forward-reference difference only |
| Guaranteed reclaimable | 0 | 0 | 0 | conservative lower bound |

NAR bytes are serialized store-object sizes, not allocated filesystem blocks.
The two exclusive rows are neither guaranteed lower bounds nor proven upper
bounds on collector release:

- daemon =gc-keep-derivations= behavior can introduce reverse deriver
  liveness that ordinary requisites do not traverse;
- running processes and temporary roots can retain paths not present in the
  persistent-root snapshot;
- hardlinks, block rounding, filesystem metadata, and collector-time state
  determine actual freed blocks; and
- root and store state can change after any point-in-time report.

The 49,072-byte difference between the policy-only and persistent-root rows
shows one additional forward-referenced path retained by another known root.
It does not improve the zero guaranteed-reclaim lower bound.

* Stop boundary and handoff

P5.1 stops before generation deletion.  Publication completed only this
read-only slice.

P5.2 must:

1. start from a clean, upstream-equal checkout and fresh estimator snapshot;
2. present exact generation numbers again and obtain explicit attended
   destructive authorization;
3. delete only reviewed generation links and collect no store object;
4. prove current, booted, pins, generation counts, =verify=, and
   =rollback-drill= afterward; and
5. stop for review.

P5.3 remains separate.  It must use a fresh post-P5.2 non-mutating estimate,
must not silently use =guix gc --list-dead=, and needs a distinct explicit
authorization before any potentially mutating discovery or collection step.

The private Org sync/history idea, backup/storage deferral, Home hygiene, Git
hygiene, and filesystem reorganization remain separate roadmap work.
